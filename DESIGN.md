# JoJo スライドデザイン・ルール / Slide Design System

このデッキ（Slidev・ダーク × cyan/purple ブランド）の**デザイン規律**と、
**ゼロから同じ品質のデッキを AI に作らせるための指示テンプレート**をまとめたもの。

「作り直し」ではなく「規律の上書き」が前提。ブランド（配色・フォント・トーン）は不変、
**余白・タイポ・整列・装飾の規律**だけを底上げする思想で書いている。

### 出典（このルールの根拠）
- koenigwolf「AIでスライドを作るためのプロンプト設計大全」 https://zenn.dev/koenigwolf/articles/b7e6874084c0da
- ai_biostat「AIスライドの“体裁を整える”を NotebookLM に任せる」 https://note.com/ai_biostat/n/nbab45053620c
- Anthropic 公式 Skills（`pptx` の "Design Ideas" 節 / `frontend-design`） https://github.com/anthropics/skills
- kappa「AIスライド作成プロンプト × デザインの4大原則（CRAP）」 https://kappa-kobuta.com/ai-slide-presentation-prompt/

要点（各記事の共通項）:
**「本質（メッセージ・構成）は人が作り、体裁（整列・余白・サイズ・色）は規律で統一する」**。
そして「1スライド1メッセージ／余白を広く／見出しは結論型（結論ファースト・PREP）／色数を絞る／役割ごとに文字サイズを全スライド固定／意味のない装飾を避ける／デザインの4大原則（近接・整列・コントラスト・反復）を守る」。

## 0. このルールの使い方（再利用と継続更新）

このドキュメントは**他プロジェクトでも同じテイストのデッキを再現するための資産**。
「DESIGN.md ＋ style.css ＋ 主要コンポーネントを持ち込み、ブランド値だけ差し替える」だけで同等品質に到達できる状態を保つ。

### 0.1 継続更新のルール（徹底）
- **ユーザーのフィードバックは“その場の修正”で終わらせず、必ず一般化してこの DESIGN.md に追記する**（個別の直し → 再発防止の規律へ昇格）。1つ直したら「同じ轍を踏まない規律」を1つ書く。
- 追記先の判断: 普遍原則 → §1–§8 ／ 再利用・テンプレート → §0・§9・§11 ／ このリポジトリ固有の実装 → §10。
- 「あえて残す」逸脱は §8.1 に**根拠つき**で記録（新規エージェントが“違反”として戻さないため）。
- コミットメッセージに**何の教訓を規律化したか**を1行残す（例: `docs(DESIGN): …の教訓を§3に規律化`）。
- この §0.1 自体が標準指示。impeccable 等のエージェントは作業前に本書を読み、フィードバック反映を毎回行う。
- **変更後の検証は「overflow=0・console エラー0」だけで終わらせない**＝必ず**3軸の設計規律レビュー**を実測で行う: ①**改行**（§4・§116–119: 行頭/語中切れ/半端な読点行/着色・引用句の分断を `Range` で行分割スキャン。新規テキスト枠は追加分も対象）②**フォント統一**（§3: 同じ役割＝同じ字体。mono キッカーは tracking/weight を既存に合わせる）③**同一ページ内で配置を変えない**（§5: カード上辺・本文 top を measure して Δ=0／タブ切替で大パーツが動かない／グリッドは全カード同待遇）。機能 QA（overflow/error/ターゲティング）で満足して**この3軸を飛ばさない**（教訓＝Phase2 で機能検証のみ実施し 改行/フォント/配置不変を明示レビューしていなかった＝ユーザー指摘で発覚）。
- **配布 PDF の同期も検証の一部**＝レンダリング結果に影響する変更（`slides.md`／`style.css`／`components/`）を入れた回は、検証の最後に **`npm run export:handout`** で `handout.pdf` を作り直す（派生成果物・gitignore・保存ごとの自動再生成はしない＝§10）。デッキと配布物が食い違わないようにする（教訓＝ハンドアウト導入時に「編集→再生成」の連結を明文化しておらず、PDF が古くなる恐れをユーザー指摘で発覚）。
- **presenter ノートは“読者も読む説明書”として書く**（登壇者向けの進行台本＝“軽く流す”等の号令ではない）＝各スライドを説明体で詳説する（1枚150〜350字目安・やさしい語り口・進行指示を書かない）。実装規律は §10（教訓＝当初のトーク台本を“閲覧者も読む詳細説明”へ位置づけ直し＝ユーザー指摘で全31枚を全面リライト）。

### 0.2 移植のしかた（ポータビリティ階層）
| 階層 | 内容（節） | 新規プロジェクトでの扱い |
|---|---|---|
| **A 普遍** | 4大原則・タイポ階層・余白・レイアウト・結論型見出し・スロップ回避（§1,3,4,5,7,8） | **そのまま適用** |
| **B ブランド** | 配色トークン・フォント・キャンバス寸法（§2） | **値だけ差し替え**、構造は維持 |
| **C 実装** | コンポーネント名・CSSクラス・ファイルパス（§10,§11） | **資産を持ち込む**か同等を再実装 |

### 0.3 新規プロジェクトで同テイストを再現する手順
1. `style.css` をコピーし、`:root` のブランドトークン（A/B 色・フォント）だけ差し替える（構造・クラス名は変えない）。
2. §11 の再利用アセット（`RevealTabs` / `Glossary` / `global-bottom` の `.gterm` / `setup/main.ts`）を持ち込む。
3. 本 DESIGN.md を置く（エージェントが自動で読み、A 普遍規律を適用）。
4. §9 のテンプレートで構成案 → スライドを生成し、最後に目視 QA を最低1周。**Pages 公開するなら `--base /<repo>/` でビルドした成果物でも、矢印/クリック送り・用語ジャンプ・リロードを1周ナビ検証する**（`setup/main.ts` の持ち込み漏れは dev では露見せず、base-path build で初めて二重 base→404 として現れる＝§10。この検証ゲートが無いと移植時に黙って落ちる）。

## 1. デザインの4大原則（CRAP）— すべての土台

> 出典: kappa 記事。スライドの良し悪しは「センス」ではなく、この4ルールを守るかどうかで決まる。
> AI に作らせる／直させるときは、毎回この4語を指示に入れる。

| 原則 | 意味 | このデッキでの具体 |
|---|---|---|
| **近接 / Proximity** | 関係する要素は近づけ、無関係な要素は離す。間隔そのものが「グループ」を語る。 | 見出し↔eyebrow は近く、eyebrow↔本文ブロックは離す。カード内は見出し→本文→キャプションを密に、カード間は16px以上あける。 |
| **整列 / Alignment** | 見えない線で揃える。左揃えを基本にし、中央寄せと混在させない。 | 本文・li は左揃え。中央寄せは表紙／章見出しのみ。カラム・カードの左端／上端を揃える。 |
| **コントラスト / Contrast** | 重要度の差を、サイズ・太さ・色で明確に。中途半端な差はしない。 | タイトルは本文の約2倍。キーワードだけグラデ／太字。muted は補助情報に限定。 |
| **反復 / Repetition** | 同じ役割の見た目（位置・サイズ・色・余白）を全スライドで統一する。 | 役割→スタイルを CSS クラス（`.eyebrow` `.card` `.grad` 等）で固定。スライドごとに変えない。 |

- 迷ったら「**この差は 近接／整列／コントラスト／反復 のどれで説明できるか?**」と自問する。説明できない装飾は削る。

## 2. ブランド（不変 / 変更禁止）

| 要素 | 値 |
|---|---|
| 背景 | ダーク `--bg-0: #07080d` → `--bg-1: #0b0d15`（上下グラデ）＋ 四隅の cyan/purple radial ＋ 微細なグリッド |
| アクセント | cyan `--brand-a: #22d3ee` → purple `--brand-b: #a855f7` の 90deg グラデ（`--brand-grad`） |
| 文字色 | 主 `--ink: #e9edf6` / 副 `--ink-soft: #c6cdda` / 補助・出典 `--muted: #8b93a7` |
| 罫線 | `--line: rgba(255,255,255,.08)` |
| 見出し・本文 | **Noto Sans JP**（weight 400/500/700/900） |
| ラベル・数値・コード | **JetBrains Mono** |
| キャンバス | 980 × 552（16:9）、外周 padding **3rem** |

- グラデ文字（`.grad`）は**見出しキーワード/ラベル/装飾のみ**。本文・li には使わない。
- 1スライドにグラデの「主役」は1つまで。
- **暖色ティア（amber `#fbbf24`＝いま/主役・rose `#fb7185`＝注意/危険）は“意味を運ぶ少量アクセント”として許容**（§6）。地の cyan/purple グラデは不変＝暖色は now/caution マーカーにのみ・控えめ（≤~10%）に差す（殺風景回避）。

## 3. タイポグラフィ階層（役割ごとに全スライドで固定）

> 最重要ルール（note 記事）: **同じ役割の文字は、全スライドを通じてサイズ・太さを完全に統一**する。
> 「このスライドだけ入りきらないから小さくする」は禁止。入らなければ**分割**で対応。

| 役割 | フォント / ウェイト | 目安サイズ | 配置 |
|---|---|---|---|
| 表紙タイトル `.cover__title` | Noto Sans JP 900 | 3.35rem | 左揃え |
| 章タイトル（`layout: section` の `# h1`＝`.section__title`） | Noto Sans JP 900 | 3.2rem | **左揃え**（文脈ラベル→タイトル→リード→route／右下に背景 `.section__chno`） |
| ヒーロー命題 `.bigstate`（1スライド1メッセージ＝式・結論） | Noto Sans JP 900 | 3.15rem | **中央** |
| コンテンツ見出し（`# h1`） | Noto Sans JP 900 | 1.7rem 前後 | 左揃え |
| カード見出し `### h3` | Noto Sans JP 700 | 1.05rem 前後 | 左揃え |
| 本文・箇条書き `p / li` | Noto Sans JP 400 | **最低 1rem** | **左揃え** |
| 結論・要点（takeaway）`.tk` | Noto Sans JP 400〜700 | 1.0〜1.18rem | **中央**（スライドを締める1行） |
| ラベル `.eyebrow` | JetBrains Mono 800・大文字・トラッキング広め | 0.72rem | 役割で固定 |
| キャプション・出典 | `--muted` | 0.7〜0.85rem | muted |

- タイトルと本文で **約2倍**のサイズ差をつけ、階層を一目で分からせる。
- 強調（bold）は **1スライド1〜2箇所**まで。多用は階層を壊す。
- 中央寄せは**タイトル / 章見出し / 結論行（takeaway `.tk`）のみ**。本文・箇条書きは必ず左揃え。
  - 結論行は各スライドの締め（要点・主張）として末尾に置き、中央寄せ＋`text-wrap: balance` で短く美しく改行する。
- **図・タイムライン・同心図の中の同種ラベルも字体を全層・全項目で統一**（フォント/サイズ/ウェイト/不透明度）。1要素だけ class 未適用で本文字体（大きい sans・不透明度1.0）に漏れると、そこだけ過剰に目立ち階層を誤読させる（実例: 入れ子図の最内 PROMPT が他ラベルより 57% 大きく描画され「最重要」に見えた）。図内で強調したい層があっても**サイズ差ではなく着色など意図した1手段**で示す。
- **mono キッカー系ラベルは字体を1つの言語に統一する**（`.eyebrow`／`.card__ba`（before/after キッカー）／`.author__meta small`／`.cover__num`／`.row__tag` 等）＝**JetBrains Mono・大文字・weight 800・トラッキング .22em** に揃える。新しいキッカーを足すときは独自値を発明せず**既存の tracking / weight に合わせる**（実例: `.card__ba` の letter-spacing を .2em→.22em に統一して `.eyebrow` と一致）。「同じ役割＝同じ字体」（§3 冒頭ルール）はモノラベルにも等しく適用。
- **ディスプレイ・スケールで“抑揚”をつける（平板なタイポは「未完成＝40%」に見える）**＝本文と見出しが同じ調子で並ぶと、内容が正しくても未完成に見える。是正は**役割の両端を思い切り離す**——(a) **表紙/章扉のディスプレイ見出しを大きく**（`.cover__title` 3.35rem／`.section__title` 3.2rem・`letter-spacing:-.005em`・`text-wrap:balance`）、(b) **章扉は左寄せの一筆書き**＝短い文脈ラベル→solid cyan の要点語を含む主見出し→リード→章内3ステップの route と進み、中央スポットライト・冗長な `CHAPTER 0X` eyebrow・gradient text を使わない、(c) **章番号 `.section__chno` は右下の背景ウォーターマーク**（低コントラストの輪郭＋薄い塗り、`aria-hidden="true"`）として残し、主見出し・ダイヤ印・route と競合させない、(d) **“1スライド＝1メッセージ”の命題は `.bigstate`**（3.15rem・中央・weight 900）。**サイズを上げるのは“主役1つ”だけ**。**letter-spacing 下限は −.04em**。検証は実寸スクショで「主役が際立ち脇役が引く」抑揚と、5枚すべての正しい章番号・h1-top・overflowを確認する。
- **言い換え・比喩をつなぐ `＝` の直前に句読点（。）を置かない**（ユーザー指摘「p12 の無駄な句読点。は不要」）＝「取得するだけ（手元は変えない）**。**＝ ポストを見るだけ」の 。は、文をいったん切ってから ＝ で言い換える**二重表現で冗長**。`説明 ＝ 比喩` と直結する（＝ は連結子＝そこで文意が続くので 。で切らない）。別の主張へ移る本当の文境界の 。は正当＝**削るのは「＝ の直前」の 。だけ**（`＝` 前後は半角スペースで呼吸を作り、句点は使わない）。実寸スクショで `）＝` `語 ＝` の連なりが窮屈でないかも確認。

## 4. 余白・間隔

- スライド外周は**最低 48px（3rem）**。端ギリギリ配置・フッター/出典と本文の衝突をなくす。
- 要素間ギャップは **0.5 / 1 / 1.5rem の3段スケール**に統一（`mt-4` / `mt-6` 等を混在させない）。**`cols eq-title` の中で見出しが2行に折り返すカードが1枚でもあると行全体が高くなり、密スライドでは最下部の `.src` が 1px だけはみ出すことがある**＝是正はスケール内で（例: カード行を `mt-4`→`mt-2` に詰める）。スケール外の半端な margin を足さない。1px でも overflow0 がデッキの不変条件。**さらに `cols eq-title` の見出し高さ予約（`.card h3 { min-height }`）は“最も折り返す見出し”を完全に含む値にする**＝2行に折り返す見出しが予約（例 2.5em）をわずかに超えると、そのカードだけ本文の開始 Y が数 px 下がり**同一ページ内でカード本文がズレる**（実例 /18「Mixture-of-Agents」が2行で bodyTop +5px）。是正＝`min-height` を最長（2行）見出しを含む値へ上げ（例 2.5em→2.75em）、**全カードの本文 top（bodyTop）を measure して Δ=0** を確認する（§5 同一ページ配置不変）。
- **1スライド1メッセージ**。詰め込まない。各スライドに意図的なホワイトスペースを残す。
- **高密度本文は文字を縮めず、意味のある工程・時間軸・判断単位でchunkingする**。同格8セルのように走査負荷が高い場合は、原因・結果・出典を落とさず「設計前／実行中／運用」の縦stackへ再編し、各行の先頭ラベルで読み順を作る。結論型見出しと締めの1メッセージを先に固定し、カードを増やして密度を隠さない。
- **メリハリ（強弱）＝「重要語を目立たせる」より先に「脇役を畳んで可視ノイズを減らす」**（ユーザー指摘「1スライドに文字が多く重要情報が把握しにくい。せっかく HTML なので出典はマウスオーバーのポップアップで／文字サイズや画像でメリハリを」）。手順 ——①**常時可視でなくてよいテキストは畳む**＝出典の羅列は `Cite.vue` の「🔗 出典」ホバー・チップに（§166/§5）＝1 スライドの可視文字数が減り、残った本文が相対的に際立つ（HTML の利点＝必要な人だけ hover で開く）。②**強調は“サイズ一辺倒”に頼らない**＝h1-top/overflow を守るため字を大きくし過ぎず、**ウェイト＋暖色アクセント**（`.hot`/`.mk` の amber＝§6）で「いま／主役」を光らせる（サイズを上げると密スライドで詰まる＝§111）。画像・イラストで語らせる所は思い切り大きく（§5 マスコット/二領域）。**“全部を大きく”は強弱にならない**＝主役1つを決め、脇役（出典・補足）は畳む・小さくする。
- **「意図的なホワイトスペース」と「中身のない箱」は別物**。余白はグループ間の呼吸であって、枠（パネル・カード）の中の空きではない。パネル/カードは**中身の量に合わせて算出**する＝縦は中身にハグ（`flex:0 1 auto`／側カードは `align-self:start`＝top ピン。タブ等で高さが変わる側カードを `align-self:center` にすると中身高が変わるたび再センタリングして“移動”する＝同一ページでレイアウトを変えない原則§5に反する）、横は短文を全幅の箱に入れない（比較カードは `max-width` で内容相応に絞る）。余ったスペースは**グループ全体を中央寄せして上下マージンに分散**させ、枠内の空白や間延びにしない。「埋めるためにパネル/カードを大きくする（全高フィル・全幅）」は過修正（§5）。measure で「枠＝中身＋パディングのみ／空箱なし／上下マージン均等」を確認。**⚠️ 例外＝タブ切替で内容が入れ替わるリビールの側カード／本文（`.rt__card`／`.rt__lead`）**: ここは no-shift（配置不変）を最優先し**“最も高いタブ”分の高さを予約**するため、短いタブには下端に空きが出る（§163）。これは静的1枚スライドの「中身のない箱」とは別物＝**意図的に許容**（“空箱なし”ルールをこの予約に対して適用して戻さないこと）。

- **矢印連結・複合語は折り返しで分断しない**: `A→B→C` のように矢印で繋いだ連なりや、スペースを含む複合語（例:「MCP サーバ」「Few-shot 学習」）は `whitespace-nowrap` で1かたまりに保護する。日本語は文字間どこでも改行されるため、放置すると**行頭が「→」で始まる**／**複合語が行を跨いで割れる**といった不自然な改行が出る（`word-break: auto-phrase` でも矢印・固有複合語までは守りきれない）。長い連なりは短い意味単位ごとに nowrap 化し、全体が次行へ送られても語の途中で切れないようにする。QA は実レンダリングを見て行頭・分断を確認する。例示の「A→B→C ＝ まとめ」型は、**動作連鎖**と**まとめタグ（＝…）をそれぞれ別の nowrap 単位**にして、折り返しが要っても**両者の境界**で改行されるようにする（タグが語の途中＝「有能な／アシスタント」のように割れない）。隣り合う対の例（LLM 行／エージェント行）には同じ保護を当てて体裁を揃える。

- **カード／ノート内の改行は意味境界で**: `RevealItem.card` やパネル `.note`・吹き出しのように幅の狭い枠内の文は、矢印（→）や区切り（／・＝）が**別の節を始める位置**で `<br>` を入れて明示改行する（例:「『資料をまとめて』<br>→ 何を調べ…」）。放置すると枠幅依存で**語の途中**（例: それ｜ら・コス｜ト）や**節の途中**で割れる。逆に `.grad` 等で**着色した語句は折り返しで分断されると見栄えが悪い**ので、その句だけ `whitespace-nowrap` で1かたまりに保護する（着色句は割らない＝折り返しは着色句の前後で起こす）。**`<strong>` で強調した中核語や、引用符（“ ”）・鉤括弧で囲んだキーワードも同様に `whitespace-nowrap` で保護する**＝日本語は文字間どこでも改行されるため、放置すると引用語が行を跨いで割れる（実例: 新規 Rubberduck スライド `.note` で “壁打ち” が「“壁｜打ち”」と語中分断＝不合格 → `<span class="whitespace-nowrap">“壁打ち”役</span>` で是正）。**新しいスライド／ノート／カードを追加したら、コミット前に必ずそのテキスト枠を実寸レンダリングして改行スキャンする**（既存スライドだけでなく追加分も §116 の対象）。QA は実レンダリングで「行頭・語中切れ・着色/強調/引用句の分断」を確認する。
- **改行品質は「overflow=0」では測れない＝各行の行頭を必ず見る（レビューの徹底）**。はみ出しゼロでも語中・節中で割れていれば不合格。目視に頼らず**自動スキャン**する＝各テキスト枠を `Range` で実レンダリングの行に分割し、**行頭が「閉じ括弧・句読点・中黒・助詞単独・`<br>` を伴わない `→`／`／`」で始まる行**を全スライド一括検出（例外: 明示 `<br>` 直後に意図的に行頭へ置いた `→` は正＝ソースで `<br>→` を確認）。さらに**狭い枠は1行の収容字数を実測**してから割り当てる（例: cols-3 カード ≈ 全角13字/行・幅広 `.row__body` ≈ 35字/行）＝各 `<br>` 区間を実測上限の**内側**に収める。1〜2字オーバーで意図しない二次折返し（`auto-phrase` が手前の文節境界で割る）が出たら、**末尾の読点（、）を句点（。）化／削る・語を1つ詰める**で1字減らす（例:「…コードが向上し、」14字→「…コードが向上。」13字で2行に収束）。修正後は必ず再レンダリングして全行頭を再確認（最低1周）。
- **強制 `<br>` は「行が埋まっていない読点行（半端な短い行）」を作らない**＝これも“不要な改行”として不合格。`<br>` を読点（、）や連用（〜し）の途中に置くと、その行が枠幅の半分ほどで切れて右側に大きな空きが残り、読者には「なぜここで改行?」と映る（実例: p9「ツールが"能力"そのものなら、」が約50%幅で停止）。**原則の優先順位**: ①まず**割れていた語句を `whitespace-nowrap` で保護**し、**自然折返しに任せて行を上端から充填**する（例: 「結果を返す」を nowrap 化→直前の文が行を埋め、句の途中で割れない）。②それでも2行になるなら `<br>` は**文末（。）の境界**＝2文を2行に分ける位置にのみ置く（読点ではなく句点で割る／各行が文として完結。実例: p12 Harness リード「素のモデルは「考える」だけ。<br>実際に動く足場…が要る。」で末尾の孤立「要る。」を解消）。③枠に**1行で収まる長さ（実測上限内）なら詰めて1行**にする（例: p9 MCP 橋渡し文を「枠組みを越えて」を削り ~29字で1行に）。判定基準＝**行末が読点・連用で、かつその行が枠の ~70% 未満の幅**なら半端な強制改行とみなして上記①〜③で是正。`Range` 行分割で各行の幅%（最長行比）を測り、最終行と見出し/ラベル行を除き **70% 未満の中間行がゼロ**であることを確認する。**ただし最終行の除外は“分割できない単文の短い末尾”に限る**＝文中に意味区切り（・／）があり、そこで割れば左右が均衡するのに auto-phrase が末尾に半端な短行を残した場合は **orphan を不合格**とする（実例: p12 リード「文脈窓に、適切な情報とツールを・適切な形式とタイミングで／**入れる設計。**」で "入れる設計。" が孤立）。是正＝**区切り（・／）の位置に `<br>`** を入れて2行を均衡させる（「適切な情報とツールを・<br>適切な形式とタイミングで入れる設計。」）。**検証対象はパネル左の本文（`.rt__lead`）も含む**＝見出し・カード・ノートだけでなく**リード本文も行頭・行末・均衡を必ずスキャン**する（今回の見落とし＝検証スコープに lead を入れていなかったのが甘さ）。**スキャン対象の日本語テキスト枠は“漏れなく列挙”する**＝本文・見出し・カード・ノート・`.rt__lead`（パネル本文）に加え、**`.section__lead`（章扉リード）も必ず含める**（章扉は装飾的で見落としやすいが、本文と同じ改行規律の対象）。
- **`word-break: auto-phrase` は日本語テキストを載せる“全コンテナ”に掛ける**（`<p>`/`<li>` だけでは不足）。文節境界で改行して**語の途中で割れる**のを防ぐこの指定は、カード本文・ノート・パネル見出しなど `<div>`/`<span>` にも必要＝未適用だと「引き／出す」「足りなく／なっていく」「良／い」「大き／な」のような語中改行が出る（`.rt__badge`・`.rt__card-b`・`.rt__card-n`・**`.section__lead`（章扉リード）** 等にも付与する）。**⚠️ 章扉のリード（`.section__lead`）は `text-wrap:balance` だけ当てて `auto-phrase` を付け忘れやすい盲点**＝未適用だと「次の一／歩へ」のように**語中で割れる**（ユーザー指摘「『要点と次の一歩』の改行は気にならないのか」＝章扉 /24 で「一歩」が分断＝不合格）。是正＝`.section__lead` に `word-break:auto-phrase` を付与し、さらに意味の塊（「次の一歩へ」）と引用キーワード（“動くもの”／“使えるもの”）を `whitespace-nowrap` で保護＝**句点（。）境界で2文が2行に割れ**、複合語も引用句も割れない。さらに**見出し中の着色句（`<em>`）は `white-space:nowrap` で1かたまり**にして、`auto-phrase` でも割れる「知っている／か」型の分断を防ぐ（着色句は前後で折り返す＝上の nowrap 規律と同じ）。原則: **新しいテキスト枠を作ったら `auto-phrase` の適用漏れを必ず確認する**。慣用句・複合語（例:「使い物にならない」）が枠端で割れて意味の塊が切れる場合は、`auto-phrase` でも防げないので**語を選び直す**（例:「実力を出せない」へ言い換え）。
- **生 HTML の inline 要素（`.nw` 等）の“先頭”に置いたマークダウンのバックティック `` `x` `` は `<code>` に変換されず、バックティックが生のまま＝“シングルクオート様の記号”として残る → `<code>x</code>` と直書きする**（ユーザー指摘「p12 の `'fetch'` のようなシングルクオート」＝`<span class="nw">\`fetch\` ＋ 合流</span>` が変換されず ``\`fetch\``` が字面に残った）。markdown-it は **inline HTML 開きタグ直後**のバックティックを code-span として認識しないことがある（同じ `.nw` 内でも、先頭でなく `**` や別 span を挟めば変換される＝再現条件は“開きタグの直後”）。**規律: `.nw` など生 HTML span の中でコードチップを出すときは `` `…` `` に頼らず `<code>…</code>` を明示**する（デッキ内の他の `fetch`／`git status` チップと字面・字体を一致させる＝“クオート様の浮き”を作らない）。検証：実寸スクショ＋対象要素に `querySelector('code')` が在るか（バックティックが残っていないか）を確認。
- **同根：ブロックレベルの生 HTML 容器（`.note`／`.tk`／`.pillar`／`.figcap` 等の `<div>`）の中では markdown インライン記法（`**太字**`・`` `コード` ``）自体が処理されない → 太字は `<strong>`、チップは `<code>` を直書きする**（ユーザー指摘の隣で発見＝p17 の `.note` 内 `` `8abd985`→`7680301` `` がバックティックのまま生表示。同じ div の `<strong>ID が変わる</strong>` は効くのに、コードだけ markdown に頼って未変換＝“HTML と markdown の混在”が事故のもと）。**素の段落・markdown リスト項目（`- …`）の中**なら `` `x` `` は正しくチップ化される（p2 の `reset --hard`・p8 の `7680301` は正常）＝**再現条件は「生 HTML ブロック div の内側」**（上の inline span 版と同じ“raw HTML 内は markdown が走らない”現象の block 版）。規律：`.note`/`.tk`/`.pillar` 等の中は**最初から `<strong>`/`<code>` で統一**（デッキの既存例＝`.tk` 内 `<code>behind</code>` に倣う）。検証：これらの容器を含むスライドは実寸スクショで**バックティック/アスタリスクが字面に残っていない**ことを確認。
- **カード内の従属ラベル（英字モノ romaji・タグ等）はカード内側端に密着させない＝クリアランスを予約する**（レビュー指摘「/22 のパターンカードで英字ラベルが枠右端に迫って窮屈」）。カード見出しが「主（日本語名）＋従（英字モノ）」の横並びのとき、**JP名も EN も最長**の1枚では従ラベルがカード右内側端まで伸びて“貼り付く”。**⚠️ ここで `flex-wrap` 有効なラベル行に単純に右パディングを足すと、その最長カードだけ従ラベルが2行に折り返して高さが不揃いになる**＝並列グリッドは高さを揃える原則（§5 同一ページ配置不変）に反する。是正は**折り返しを起こさずにクリアランスを作る**こと＝従ラベルのフォントを **system 内の最小モノ値**へ一段下げる（新しい値を発明しない）。主従の階層（JP=主・EN=従）も明確になり一石二鳥。**実例**＝/22 `.ptn` パターングリッドの最長カード「オーケストレーター–ワーカー／Orchestrator–Workers」は EN 右端がカード内側右端の残り **12.4px** まで迫っていた（他3枚は 147–302px 余裕＝1枚だけ突出して密）。`.ptn__en` を **.66rem→.60rem**（＝`.author__meta small`／MCP ラベルと同じ既存の最小モノ値）に下げ、EN 幅 170.8→155.3px・内側クリアランス **12.4→27.9px** に回復。全4枚 `nameLines:1` 維持＝折り返しゼロ・高さ揃い。**検証**＝従ラベルの右端と親カードの content-box 右端（`rect.right − paddingRight`）の差を measure し、最長カードで**左パディング以上のクリアランス**があること、かつ**全カードのラベル行が1行**（`height/lineHeight≈1`）であることを確認（フォント縮小は縦を増やさない＝§117 の h1-top 詰まりの副作用なし）。**やってはいけない**＝(a) ラベル行に右パディングだけ足す＝最長カードのみ wrap して高さ不揃い（§5 違反）／(b) 主（日本語名）側を縮める＝主従が逆転／(c) system 外の半端なサイズを新設（既存の最小モノ値を使う）。
- **狭いportrait面は「本文幅・図幅・区切り線」を別々に最適化する**（ユーザー指摘「文章がものすごく改行／絵の幅が無駄に大きい／区切り線が多い」）。①**本文幅**＝短いラベル・lead・takeawayは`text-wrap:balance`で孤立尾を避け、通常viewportでは構造テキスト（label/list/visual label）を原則3行以内、全可視本文を4行以内、最終行が最長行の12%未満になるorphanを0にする。超えたら文字を16px未満へ縮めず、アイコン／rail列を細くして本文へ幅を返すか、二列を全幅stackへ戻す。`Range`は要素全体だと子spanのboxを二重計上するため、**text nodeごとのclient rectを同一top±2pxで束ねて**測る。②**図幅**＝meaningful visualの判定は面積の大きさでなく意味構造。情報量の少ないroute/target/ring図を反射的に全幅へせず、内容相応（pilot detail図はstage幅76〜84%、上限86%）に絞って中央配置する。③**区切り線**＝近接・余白・bulletでgroupingできるlist itemへ上下borderを反復しない。残す線はtimeline spine、decision fork、lifecycle connectorなど**関係や順序を運ぶ線だけ**。QAで対象listのcomputed borderを検査し、罫線の再導入をfailにする。

## 5. レイアウトパターン（使い回さず変化させる）

全スライドに**視覚要素（図／カード／チップ／タイムライン／アイコン／表）を1つ以上**。文字だけのスライドを作らない。

- **地の“奥行き・質感”を作る（平坦な近黒は「安っぽい＝40%」に見える主因）**＝背景が均一な near-black のままだと、内容が良くても未完成に見える。ブランドの dark と cyan/purple を**光源**として使い、`.slidev-layout` の背景を**4層に積む**——(1) 左上から差す cyan グロー（`radial 1080×560 at 4% -16%`・brand-a 22%）(2) 右下からの purple グロー（`at 116% 122%`・brand-b 21%）＝**cyan→purple のブランド対角を“光の軸”**に、(3) 軸から外れた隅を沈める淡いヴィネット（`radial 145%135% at 50% 32%`・#000 58%）＝内容が浮く、(4) 既存のベース縦グラデ。さらに `::before`＝**設計図グリッド**（46px・不透明 .5・左上へマスク＝技術テクスチャ）、`::after`＝**微細フィルムグレイン**（グレースケール `feTurbulence`・不透明 **.05**・160px タイル）で大面積のフラットな暗部が“死んだ黒”に見えるのを防ぐ。**content は `z-1`／擬似は `z-0`** なので文字は常にグレインの上でクリップ（可読性不変）。これは§8 が禁じる「過剰な glow/影」**ではない**＝不透明度を実測レンジ（グロー ≤22%・グレイン 5%）に抑えた calibrated な地（§8.1 に例外登録＝平坦単色へ戻さない）。**カード/パネルは“持ち上がった面材”にする**＝`--surface`（上が明るい `linear-gradient(180deg, rgba(255,255,255,.052), .016)`＝上方光源と整合）＋`--surface-shadow`（`inset 0 1px 0 rgba(255,255,255,.06)` の内側ハイライト＋`0 14px 34px -22px rgba(0,0,0,.9)` の柔らかい落ち影）を `.card`/`.row`/`.pillar`/`.ptn` に適用＝平板な塗り面を**光を受けた立体パネル**に。**2値のトークンなので移植は `--surface*` の差し替えだけ**（§11）＝`.pillar.b`／`.row.accent` の色替えは border/tint だけを上書きするので共存する。検証は実寸スクショで p1/p5/p9 等が「浮いて見える／文字は crisp」を目視（overflow0 では奥行き不足を検出できない）。

- **縦並びパネル（アイコンレール + 本文）** `.stack` > `.row`: 比較・列挙を**横に潰さず縦に積む**。狭いカラムでの「文章の折返し」を防げるので、テキスト主体はこれを優先。`.row.tight` は1行説明用のコンパクト版、`.vsbadge` は比較の接続子（VS / 〜↓）。
- **2カラム比較**（左右並列）: 違い・Before/After・単一 vs マルチ など（図解・カード・短文向き。長文は縦並びへ）
- **アイコン + テキスト行**: 理由・特徴の列挙
- **2×2 / 2×3 グリッド**: 4性質・落とし穴・**型／パターンの一覧**など。**「N個に集約」と銘打ったら N個すべてを同じ粒度で説明する**＝代表1つだけを図示して残りを名前チップだけ、にしない（各項目に アイコン＋名称＋英名＋一言説明＋例 を与え「説明不足」を解消）。1パターンを詳述する隣接スライドがあるなら、この一覧は**全パターンの俯瞰**に徹する（重複させない）。
- **タイムライン / フロー**: 手順・ループ（ReAct 等）・進化の流れ
- **図解（mermaid）**: 木構造（オーケストレーター）・入れ子・関係図（`max-height: 430px`）。**デフォルトのグレー箱のまま置かない**＝`classDef` でブランド配色を割り当て、他の自作コンポーネントと同じ仕上がりに揃える（役割で色分け: 認知/入力＝cyan `#22d3ee`・行動/実行＝purple `#a855f7`・判定＝amber・完了/成功＝green。枠 1.6px・暗い同系 fill・明るい文字）。ノードは「アイコン＋日本語（主）＋英語名＋一言の役割」の多段ラベルで**意味の密度**を上げ（「簡素すぎる」を避ける）、図のフェーズは**スライドのタイトルと一致**させる（例: タイトルが「知覚→思考→行動→観察」なら図も同じ4フェーズ＝知覚を補う）。ループは戻り矢印に短い理由ラベル（例:「まだ → 再試行」）を付け、概念の深掘り（例: 反復ごとに文脈が増える）は**キャプション側へ逃がして図はすっきり**保つ。長いエッジラベルは mermaid が語の途中で折り返すので短く保つ（`scale` で全体幅を 980px に収め overflow0 を measure 確認）。同じ配色規律をデック内の全 mermaid に適用し一貫させる。**gitGraph（コミット履歴図）はコミットラベルが既定で約 −45° 回転＋小さめ＝暗背景では重なって判読不能**になる（色だけ `classDef`／`themeVariables` で揃えても、ラベルが読めなければ図は死ぬ）。`setup/mermaid.ts`（`defineMermaidSetup`）に **`gitGraph: { rotateCommitLabel: false }`** を与えてラベルを水平化し、`themeVariables` の `commitLabelColor`（明色・例 `#e6eaf2`）／`commitLabelFontSize`（例 `13px`）／`commitLabelBackground: transparent` で可読性を上げる（ブランチ枠は `git0..n`／`gitBranchLabel*`、タグは `tagLabel*` で配色）。水平化はグローバル設定で全図に効く＝各図の `%%{init:{'gitGraph':{…}}}%%` は深いマージなので `showBranches`／`mainBranchName` と共存する。検証は**実寸スクショで各コミットID（`A`/`B`/`C`・短縮ハッシュ・「最新」等）が読めるか**を確認（overflow0 だけでは回転・低コントラストのラベルを検出できない）。
- **gitGraph は「id 無しの `commit`」を禁止＝全コミットに短い明示 id を付ける（＋図は主役サイズに拡大する）**（ユーザー指摘「p13 を含め基本図が小さく、コミット文字列のようなものが重なって表示される」）。id を書かない `commit` は mermaid が **`0-2061de…` のようなランダム長ハッシュを自動生成して表示**し、隣の明示ラベル（`origin/main`・`最新` 等）と**水平に重なって判読不能**になる（`rotateCommitLabel:false` で水平化済みでも、ラベルが commit 間隔より長ければ潰れる）。是正：**すべての `commit` に短い id**（`A`/`B`/`C` か 3 文字ハッシュ `a05`/`92b`）を与え、**日本語や長い文字列（`私のコミット`・`origin/main`）を id にしない**（意味は本文・キャプションで補う＝手本は単文字 id の merge 図＝重なり皆無）。“behind by N”のように**数を見せる図は N＋1 コミットに絞る**（分岐点に `local`、`origin/main` を N 個先へ＝「差」が一目で分かりラベルも余裕を持つ）。さらに **gitGraph は既定で小さい**ので図を拡大＝`.fig .mermaid{transform:scale(1.3+)}`（全幅図）／`.cols .mermaid{transform:scale(1.1)}`（2 カラムは列幅を尊重して控えめに）。検証：実寸スクショで**ランダムハッシュが出ていない・ラベルが重なっていない・図が十分大きい**の3点を確認。
- **比較表**: フレームワーク早見表 など
- **大きな数字 + 小さなラベル**: 重要な数値があるときだけ（無意味な hero-metric 量産はしない）
- **概念（比喩）スライドの図は、その比喩を“実際に描く”＝抽象チェーンで代用しない**（ユーザー指摘「p8/p9 は図も説明も Git 分かる人でも全く分からない」）。「commit＝スナップショット」「branch＝付箋（ポインタ）」のように**比喩で概念を教えるスライド**で、mermaid gitGraph の**ただのドット列**（`showBranches:false` の水平チェーン）を置いても、点が並ぶだけで「写真」「付箋」を一切可視化できず、図が本文の比喩と噛み合わない＝“分からない”図になる。規律：**図はスライドの比喩そのものを絵にする**（commit＝“その瞬間の全体像を写したフレーム”が積み上がる `.snapfig`／branch＝tip commit に貼られた“付箋 main”＋前進する ghost の `.brfig`）＋**短いキャプション `.figcap`** で「何を表す図か」を明示し、**本文・タイトル・図の3者を同じ比喩で一致**させる。**❌ 教える対象を図から隠さない**＝branch を教える回で `showBranches:false` にしてブランチ名を消すのは自己矛盾（主題の付箋・HEAD・ポインタは図に**明示**する）。**抽象 gitGraph はコミット履歴の“流れ”を見せる回にだけ使う**（fetch/pull/push・rebase 手順など＝そこは適切）。`.figcap` は `word-break:auto-phrase`＋比喩語（“写真”“付箋”）を `.nw` で囲み割れを防ぐ（§3 禁則）。検証：**Git 既知の人が図だけ見て比喩が読めるか**＋実寸スクショで図・本文・タイトルが同一比喩かを確認。
- **図が良くてもキャプション／ラベルで概念を殺さない＝“ポインタの間接参照”と“何が動くか”は平文で明言する**（ユーザー指摘「p9 は図はものすごく良いのに『HEAD → main → いまいる C』やその後の文の意味が分からない」）。良い比喩図に付ける説明を**矢印グリフ `→` の羅列**で済ませると、`→` が「指す（間接参照）」なのか「流れ（時系列）」なのか読者に判別できず、`HEAD → main → いまいる C` のような**英名ポインタ＋和文位置語の混成**は暗号化する。是正3点 ——（a）**間接参照は動詞「指す」で明言**＝`HEAD は main を指し、main は C を指す`（矢印に意味を推測させない／`.brhead` は `flex-wrap:wrap; justify-content:center` で chip 混じりの一文が中央で自然に折り返る）。（b）**多段ポインタは「辿った先＝現在地」と結論づける**＝`だから“いる場所”は C`（HEAD→branch→commit を追うと現在地に着く、を言い切る）。（c）**キャプションに“矛盾に見える”言い回しを残さない**＝`付箋（と HEAD）は前進` と `動くのは付箋だけ` を並べると「HEAD も動くのか動かないのか」と読める → **主語を分けて言い切る**（`main（と HEAD）が D へ進む＝動くのは付箋だけで、A・B・C は動かない`＝“動く＝ブランチポインタ／不動＝既存コミット”を対で明示）。検証：**Git 未修者が図＋キャプションだけで「HEAD が何を指し・新コミットで何が動くか」を誤解なく言えるか**を実寸スクショで確認（図の出来に安心してキャプションのレビューを飛ばさない）。
- **本文・タイトル・表・要約で概念の“代名詞”に使う記号は、必ず図の中に実描画して接地する＝図に無いグリフ（◆ 等）を概念名の代わりに使わない**（ユーザー指摘「p16『merge ＝ ◆ で2本を1つに』の◆の意味が分からない」）。◆ を「マージコミット」の代用記号としてタイトル/本文/比較表/要約ピラーに撒いても、mermaid gitGraph のマージコミットは**中空の丸（リング）**で描かれ**図のどこにも◆が無い**＝読者は参照先を探して迷子になる（“分からない”の正体は「記号↔図」の不一致）。是正：**図に実在するラベルで指す**＝マージコミットは**図の `M`**（「合流点に、2つの親を持つマージコミット M が生まれます（図の右端）」のように名指しして図↔本文を結ぶ）。記号を使いたいなら**図側でその記号を実際に描いて**初めて本文で使える（接地なきグリフは禁止）。**一度使った代用記号は全スライド横断で潰す**＝◆ は p16 だけでなく p18 比較表（`分岐＆合流（◆）`→`分岐＆合流`）・p25 要約ピラー（`◆ が残る`→`記録が残る`）にも波及していた（1箇所直すなら `grep '◆'` で全出現を一括是正）。検証：**その記号が図に実在するか**＋実寸スクショで本文の指示語（「図の M」「図の右端」）が図の要素へ正しく着地するかを確認。
- **対比を教える兄弟スライドの図は“互いに構造で見分けられる”こと＝描画ツールの既定で片方が別概念の図になっていないか（mermaid gitGraph の `merge` は常にマージコミットを描く）**（徹底レビューで発覚：p15「fast-forward ＝ 分岐なし・一直線」の図が `merge fix` を使い、**マージコミット付きの Y 字＝p16「merge」と同一形状**だった）。fast-forward は「分岐せず、付箋（ポインタ）が前に滑るだけ／履歴は一直線」なのに、mermaid の `merge` コマンドは**必ず2親のマージコミットを生む**＝図が概念の“逆”を描き、しかも merge スライドと見分けがつかない（対比が死ぬ）。是正：**fast-forward は単一レーンの一直線**で描き（`commit id:"A" / commit id:"B" tag:"main" / commit id:"C" tag:"fix"`）、ブランチポインタは**タグで位置を示す**（main は B、fix は 1 つ先の C＝「相手の真上」＝スライドすれば追いつく）＝マージコミットを出さない。原則：**A/B/C を対比する図の集合は、各図が自分の概念を構造で体現し、隣とシルエットが明確に異なること**（FF＝一直線＋ポインタ／merge＝Y 字＋合流コミット M／rebase＝土台の上へ積み直し）。`merge` を使ってよいのは**本当に合流コミットが生まれる回だけ**。検証：**兄弟スライドの図を並べて「ひと目で違う操作」に見えるか**＋各図が概念の逆を描いていないかを実寸で確認（gitGraph は概念の“流れ”を見せる回に限る＝§156）。
- **繰り返し構造のコンポーネント（章扉・カード群・要約ピラー）は全インスタンスで様式を統一する＝1つだけ要素が欠けていないか“集合”で監査する**（徹底レビューで発覚：章扉 CHAPTER 01–07 のうち **p24 だけ `.section__lead`（1 行リード）が無く**、他 6 枚と不揃いだった）。是正：全章扉に **eyebrow ＋ タイトル ＋ リード** を必ず与える（1 つだけ欠けると“作りかけ”に見える）。原則：同じ役割のスライド/パーツは **1 枚直したら同種を全部見る**（§0.1 の徹底＋§150「N 個は同じ粒度」と同根）＝章扉のリード有無・RECAP のピラー数と銘打った数（例「7 つの核」→ピラーちょうど 7）・カードの枠色規律などを**横断で一致**させる。検証：`grep 'section__lead'` 等で**出現回数＝該当スライド数**を数えて抜けを潰し、実寸スクショで全インスタンスを見比べる。
- **図専用コンポーネントのスタイルは“図のスコープ内”に閉じる＝汎用名のクラス（`.tag`/`.note`/`.card` 等）を図の都合でグローバル定義しない（表紙・共通レイアウトの同名クラスへ漏れる）**（ユーザー指摘「p27 の表紙（結び）スライドで SERIES バッジの重複が発生している」）。p9 の付箋を作るために `.tag{position:absolute; top:-34px}`＋`.tag::after`（下向き三角）を**グローバル**に足すと、表紙・結び（closing）の `SERIES` バッジ（`.cover__series .tag`＝本来は行内の pill）にも absolute 指定が漏れ、**バッジが宙に浮いてタイトルに重なる**（結びは中央縦積みなので直撃＝“重複”の正体）。同名クラスが「図の付箋」と「レイアウトの pill」の**2役**を負い、後から足した図側の規則が既存の共通コンポーネントを破壊する典型。是正：**図固有のスタイルは必ず図のルート要素へ子孫結合子でスコープする**（付箋は常に `.cdot` 内にあるので `.tag{…}`→`.cdot .tag{…}`／`.tag::after`→`.cdot .tag::after`）＝共通の `.cover__series .tag` は既定の `position:static`（行内 pill）へ戻り、表紙・結びとも `SERIES` がテキスト左に自然に並ぶ。検証：`grep 'class="tag'`（対象クラス）で**全用途を洗い出し**、図（p9）と共通レイアウト（表紙・結び）の**両方**を実寸スクショ＝図側は付箋が dot 上に三角つきで載る／共通側は pill が行内に収まりタイトルに重ならない、を同時に満たすこと。**新しく汎用名のクラスを足す前に既存の同名定義を grep**（`.tag`/`.note`/`.card` は衝突しやすい）＝「グローバルに足したスタイルが別スライドを壊していないか」を常に疑う。
- **1行だけの本文を載せる並列パネルは“高さを詰める”＝内側余白が外側余白を上回る「余白の逆転」を作らない**（ユーザー指摘「p12 はパネル内に余白がいっぱいなのに、タイトルやフッターに余白がない」）。アイコンレール付き `.row` はアイコン＋タイトルを**縦積み**するため背が高く、本文が1行だと行が上下中央でぽつんと浮いて**パネル内に死に余白**が生まれ、その分スライド全体が縦に詰まって**タイトル/締めが上下端へ貼り付く**（＝「外側の余白が無い」の正体）。是正：1行本文の並列パネルは**コンパクト版**（`.row.rowline`＝アイコンとタイトルを横並び `flex-direction:row`・アイコン縮小・パディング圧縮）で行高を本文1行相当まで下げる＝浮いた高さが**上下の外周マージン**に回り `justify-content:center` が効いて天地が均等になる。原則：**内側パディング ≤ 外周マージン**（中がスカスカで外がキツキツ、はレイアウト崩れのサイン）。`.row.tight`（パディングだけ圧縮）は頭が高いままなので、単行パネルは**頭も横並び**にする。検証：実寸スクショで「パネル内の死に余白」と「タイトル/フッターの窮屈さ」が**同時に**消えたかを目視（overflow0 では余白の偏りを検出できない）。
- **概念・操作スライドには“実行コマンドの具体例を1つ”添える＝比喩・図だけで終えない**（ユーザー指摘「各スライド、実行コマンドの具体例を記載して」）。仕組みや比喩（commit＝写真／branch＝付箋／rebase＝土台を置き直す）を説明したら、**その場面で実際に打つコマンドを1行**見せる（`git commit -m "…"`／`git switch -c feature`／`git merge --ff-only main`／`git merge feature`／`git rebase main`）＝抽象と手の動きを結ぶ。器は `.cmd`（`例` ラベル＋`code`＋任意の一言キャプション）で1行に収め、**列挙せず“最も代表的な1つ”**に絞る（早見表スライドが別にあるなら重複させない）。検証：各概念/操作スライドに実コマンドの `code` が1つあるかを確認。
- **操作スライドは「こんなとき（場面）」と「こうなる（結果）」を、仕組みと別立てで明示する**（ユーザー指摘「p17 等、どんなシナリオでその操作をするのか／実行するとどうなるのかが分からない」）。操作（fast-forward／merge／rebase 等）は**何をするか（仕組み）**だけでなく、**いつ使うか（トリガとなる状況）**と**実行後どうなるか（結果の状態）**を読者は知りたい。是正：`.uses`＞`.use.is-when`（場面）／`.use.is-then`（結果）の2行を添える（例: rebase＝こんなとき「自分専用ブランチを最新に追従させたい」／こうなる「自分のコミットが最新の上に積み直り、履歴は一直線」）。ラベルは**役割色で弁別**（場面＝amber・結果＝gold＝§6 の既定階層内）。検証：各操作スライドに「場面→結果」が揃っているかを確認。

- **並列の `.row` ペアは“左右対称のハイライト”にする＝片方だけ枠を光らせない**（ユーザー指摘「p3/p6 で、なぜ片方の枠だけハイライト? 両方同じ表示に」）: 同じ粒度で並ぶ2つの `.row`（こんな人向け↔持ち帰れる／ふつうの LLM↔エージェント 等）で、片方だけ `accent`（点灯枠）が付くと「そちらが主役・他方は従」と**意味の差**に誤読される。**両方に `accent` を付け、各行は自分の `--c` 色で発光**させる（例: cyan＋purple、免責は amber＋cyan、道具と記憶は cyan＋indigo）＝ユーザーの言う「同じ表示」＝**同じ“扱い”（どちらも光る）**であって「無装飾で揃える」ではない（殺風景§143/§179 へ逆戻りさせない・枠線がアイコン色と一致してむしろ整う）。デッキ全 `.row` ペア（免責/ゴール/チャット/道具と記憶/判断ガイド/カスタマーサポート/次の一歩）に適用。**例外＝before/after の段階対比は非対称のままが正**（例: MCP「これまで（淡い `.card`）→ MCP のあと（`.card accent`）」は“暗→明”が意味＝進歩を運ぶので片側だけ点灯を意図的に残す＝§8.1 に記録）。原則: **横並びの同格要素は反復（§58 Repetition）＝同じ役割は同じ見た目**、差をつけるのは役割が違う時だけ。

- **「遷移・進化」を見せる図は静的な箱の羅列にせず、方向と動きを持たせる**（ユーザー指摘「Prompt→Context→… の遷移図は遷移状況がわかりづらい」「他スライドにも大きなイラストや動的要素を入れてよい」）。——（a）**並列の集合（`row`＝4性質など）は連結しない**が、**順序のある系列（`timeline`＝4時代）は連続したレール**にする＝タブ間を**グラデの連結セグメント＋進行方向に流れるパルス**で繋ぐ（`.rt--timeline .rt__arrow` を作り替え＝`::before`＝レール線／`::after`＝走るパルス／`›` 文字＝矢じり）。パルスは**隙間なく連結した連続スイープ**にする＝3本の `::after` を同一周期で `animation-delay` を等分（0／T·⅓／T·⅔）し、各々は前⅓で渡り切って `opacity:0`＋`animation-fill-mode:backwards`＝**コメットが常に左→右へ流れ続け暗転なし**（前セグメントが渡り切る瞬間に次が始まる＝ハンドオフ連結）。**各セグメントを独立ループさせない**（個別ループは隣の戻り頭と重なり「矢印が戻る」ように見える＝不可）。躍動感UP: 周期は短め（≦2.4s）でコメット尾（多段 box-shadow）＋レール線自体も `background-position` で常時フロー、セグメント間 gap は無くし連結。**レール線・コメット軌道・矢じり頭は両カード端へ実測で密着させる（突き抜け禁止）**＝`getBoundingClientRect()` で各端の overshoot を測り、左右で侵入量が異なる場合は inset を**非対称**に詰める（例: 左 -1.15rem ぴったり／右 -.4rem。flex-end＋矢じりの負マージンで右が侵入しやすい）。許容 overshoot は±3px 程度＝隙間も突き抜けも目視で消える。色は cyan→中間→purple で現在＝Loop へ。検証は ::after の opacity/left を時系列サンプリングし「常時1個以上点灯（暗転フレーム無）・1→2→3 順・left 単調増加」を確認。これで「系列＝レール」「並列＝独立カード」が一目で区別でき、`row` と `timeline` の見た目を**意味的に**変える（同一カードグリッドの単調も回避）。（b）**入れ子（包含）図は「外側への広がり」を内→外のリップルで動かす**＝各リング（`.nest*`）に `box-shadow` グローのパルスを**内側ほど早い遅延**で当て、PROMPT→…→LOOP と波が外へ拡がる（締め『外側への広がりこそ進化』を運動で具現）。図自体も**大きく**（padding／ラベルを拡大して余白を埋める）。（c）**全アニメに `@media (prefers-reduced-motion: reduce)` の代替**（停止＋中庸の静止状態）を必ず添える。検証は**複数フレームのスクショ**で各段のグロー/パルスが実際に出ることを確認（単フレームだと無発光の一瞬を掴んで「動いていない」と誤判定する）。

- **スライドが「実在の OSS／プロダクト」を名指しするなら、自作の似せ絵より“公式の公開ライセンス資産”を優先してよい（ただしライセンスを実ファイルで検証・著作権表示を同梱・出典を明記）**（ユーザー指摘「ロブスターをもっと本物の OpenClaw 画像に寄せられないか／本物の画像でいい、出典を書けば」）。自作マスコット（§147）はゼロ依存で安全だが、対象が**実在の公開プロジェクト**なら**本物の公式アセット**の方が説得力・トレンド感が高い。手順 ——①**ライセンスを“実ファイル”で確認**＝リポジトリの `LICENSE` を実際に読む（MIT/Apache/CC-BY 等の許諾を確認。AI 要約や「たぶん MIT」で進めない／200 を踏んで本文を見る）。②**“出典を書けば OK”ではない**＝出典明記は著作権の**許諾ではない**。許諾を与えるのは**ライセンス**（MIT なら複製・改変・配布が可、ただし**著作権・許諾表示の同梱が条件**）。③**著作権／許諾表示をアセットに同梱**＝コンポーネント冒頭コメントに source URL＋ライセンス名＋著作権者＋ MIT 本文の要旨を書く。④**出典を可視化**＝`.src` 脚注に「マスコット: OpenClaw（MIT）」＋リポジトリリンク（§5 出典規律）。⑤**商標ロゴは nominative use の範囲**＝その対象を指す用途に限り、無関係な装飾へ流用しない。⑥**実装はインライン SVG**＝`<img src>` は GitHub Pages の base 前置で壊れやすい→ rect 等を `.vue` に直書きすれば base 非依存＋ゼロ依存＋拡大に強い。**ドット絵（pixel-art）は `image-rendering:pixelated`＋`shape-rendering:crispEdges`** で拡大してもにじませない。**本物の色は変えない**（“本物”の価値＝忠実さ）＝ブランド係留は**本体に触れず背後の cyan ハロー“だけ”**で行う（§147 の「モチーフをブランド色で塗る」は**自作イラスト限定**。公式資産は塗り替えない）。例: p16 Steinberger のカードに OpenClaw 公式マスコット（ドット絵ロブスター `#ff4f40`・MIT・© 2026 OpenClaw Foundation）を `OpenClawLobster.vue` でインライン化（`docs/assets/pixel-lobster.svg` を忠実コピー）。**※ 人物の顔写真は別扱い**＝肖像権＋多くは無ライセンス＝引き続きスクレイプ不可（§139 アバターはモノグラム）。本規律は「実在プロジェクトの公開ロゴ／マスコットで**ライセンスを検証できたもの**」にのみ適用する。

- **同一ページ内で“配置”を変えないの原則を chrome にも適用**: フッタの章ラベルは章扉でその章名を重複表示するが（例: CH05 扉の中央大見出しと下端フッタが両方「05 · 勘所とまとめ」）、**選択的に隠さず一貫して出す**（ページ番号は扉でも有用・決定論の一貫性を崩さない）。chrome を「あるスライドだけ体裁を変える」と、まさに §5「同一ページでレイアウトを変えない」で嫌う不揃いを chrome 側で作ってしまう。隠すのは**本文と物理的に重なる時（`foot:false`）と、固有ブックエンドを持つ表紙・締めだけ**に限定する。
- **カード上辺アクセントバー＝“色が意味を持つカード”にだけ付ける（無地の同格グリッドには付けない）／before/after は軽量な色付きヘッダラベルで**（参考デッキの構造をブランド不変で採用）: 参考デッキは対比パネル（TOOL/SERVICE・clone/pull・BEFORE/AFTER）を**太い左サイドストライプ**で色分けしていたが、side-stripe は impeccable の absolute ban。**天面（上辺）3px バーへ翻案**すればリボン（§額装＝3px）と同じ額縁言語で揃い、side-stripe に該当しない（§8.1）。ただし**貼るのは色が意味を持つカードだけ**＝(a) 役割色で色分けされた**セット**（例: Rubberduck /18 の LLM-as-Judge=cyan／Architect-Editor=indigo／Mixture-of-Agents=purple＝3枚を別物として識別させる）、(b) before/after の**進歩側**（例: MCP /11 の「MCP のあと」＝accent に cyan バー）。**無地の同格3カード（/9 なぜいま・/25 評価安全コスト）には付けない**＝全カード一律に付けると impeccable の「同一カードグリッド」クリシェを助長する（装飾のための装飾）。CSS は `.card.accent`／`.card.ac-*` の `::before`（`position:absolute; inset:0 0 auto 0; height:3px; border-radius:.85rem .85rem 0 0; background:var(--c,brand-a)`＋glow）で**アクセント class を持つカードにだけ自動適用**、無地 `.card` は対象外＝“意味のあるカードだけ”をセレクタで担保（貼り忘れ/貼りすぎを構造で防ぐ）。before/after ラベルは `.card__ba`（mono・.62rem・大文字）で「BEFORE（muted）／AFTER（cyan・`.ac-pu` は purple）」の軽量キッカー（参考 p05 の色帯ヘッダを軽量化）＝**1デッキ1対の意図的対比**なので eyebrow クリシェに当たらない（§8.1）。段階対比の非対称（暗→明）は §143 の対称ハイライト規律の意図的例外として維持。

整列ルール:
- カラム・カードの**端（左端・上端）を揃える**。ガター（カード間/カラム間）は **16px 以上**。
- 同列に並ぶ要素は高さ/幅を揃え、等間隔に。

出典・リンク:
- 主張の根拠は**一次情報へのリンク**で示す。スライド末尾の小さな出典注（`.src`）と、巻末の **References スライド**（`.refs` / `.refgrp`）に集約。**外部の定義・固有概念・フレームワーク名・固有手法・研究上の主張を載せるスライドは、その場に出典を必ず添える**（図解か否かを問わず＝図があることは出典を省く理由にならない。巻末 References 任せにしない）。出典の置き場は2系統 ——（a）静的スライドは末尾に `.src`（`出典: <a>ラベル</a> ／ …`）、（b）クリック式リビールは `RevealItem.sources` の**下部バー**に出す（「▸ クリック」hint とは排他）。**免除はメタ系スライドのみ**＝章扉・目次・表紙・**免責（Disclaimer）**・ゴール/対象・まとめ（要点振り返り）・用語集・References 自体。出典スイープの QA: 概念スライドを一巡し**「出典のないスライドが残っていないか」を精査**する（§0.1 のフィードバック反映の一環）。URL は追加前に 200＋タイトルを実在検証。
- **出典は native disclosure に畳む**＝常時可視の出典行は密度を上げるため、`components/Cite.vue` の **`<details><summary>`** で「出典 N件」を表示し、リンク一覧をabsolute panelとして開く。Enter/Space・focus・open状態はブラウザ標準へ委ね、hover-only／疑似buttonを使わない。静的スライドは `place="br"`（スライドの直接の子）、リビールは `place="bar"`（`curSources` 連動）を維持し、panelは本文を押し動かさない。リンクは常にDOM内に置き、`@media print` ではopen状態に依存せず展開して帰属を残す。新規Citeは右下・上部・barの各配置で本文衝突とkeyboard操作を実測する。
- **重要概念には出典を“1つで済ませない”**。各「○○エンジニアリング」など中核トピックは、原則 **2〜3 本の一次情報**（公式エンジニアリング記事＋原典論文＋実務ガイド等を混ぜる）を併記する。クリック式パネルは `RevealItem.sources: {label,url}[]` に複数渡すと**下部バーに「出典 a · b · c」**として 1 行表示（`source` 単数は後方互換、`sources` が優先）。短いラベル（org 名／著者+年）で 1 行に収め、`white-space:nowrap` で語の途中改行を防ぐ。**URL は追加前に必ず実在検証（200＋タイトル確認）**し、AI 生成の arXiv ID 等を鵜呑みにしない。巻末 References もデッキ本編の出典と整合させる。
- HTML スライドはそのまま**クリック可能なサイト**になる（Slidev ビルド = SPA）。リンクは raw `<a href target="_blank">` で**別タブ表示**にし、デッキから離脱させない。
- リンク文字はブランド cyan＋下線。表（早見表）のフレームワーク名も公式 Doc へリンク。

タイトル位置とパネル内ページング（インタラクティブ・スライド限定）:
- **スライド面はインタラクティブも非インタラクティブも `justify-content:center` で縦中央寄せ**（`:has(.rt)` でも同じ）。リビール・パネルは**幅は中身にハグ**しつつ、**高さは“最も高いタブ”分を予約**して全タブで枠を固定する。さらに**左カラムの本文（`.rt__lead`）と側カード（`.rt__card`）も各自“最も高いタブ”分を予約し、内容は上揃え（top-align）にする**。これにより**タブを切り替えても h1・パネル見出し・本文の先頭行・サイドカードの見出し/箱が1pxも動かない**（＝同一ページ内でレイアウトが変わると混乱する、というユーザー要件「配置を変えない」）。短いタブの余りは**各ブロックの下端に空きとして溜める**（＝下端の hollow は許容）。⚠️ **重要な訂正**: 旧記述「短いタブは中身をパネル内で**中央寄せ**（`--single { justify-content:center }`）するので no-shift と no-hollow を**両立**」は**誤り＝撤回**。枠の“高さ”は予約で固定できても、中身を枠内で**中央寄せすると内容の“先頭”がタブごとに動く**（中身が短いタブほど下がる＝まさにユーザーが指摘した「微妙にタイトル/配置がズレる」の正体。h1 は `<Transition>` の外で不動なので、動いていたのは**パネル本文と側カードの中身**だった）。**no-shift と no-hollow は両立しない**ので、ユーザー最優先の **no-shift（配置不変）を採り、下端の空き（hollow）を受け入れて全要素を top-align ＋ 高さ予約で固定**する（`--single` は `justify-content:flex-start`／`.rt__card` も `justify-content:flex-start`＋`align-self:start`）。旧 `flex:1` の“全高フィル”（スライド全高まで伸ばす→空箱化）とも別物＝**予約するのは“最も高いタブ”の高さだけ**。
- **最も高いタブの高さは“オフスクリーンの隠し測定スタック”（`.rt__measure`）で測る**。全タブのパネル markup を**実幅**で重ね描きし（`position:absolute; visibility:hidden; height:0; overflow:hidden`＝祖先の `scrollHeight` を膨らませず overflow を出さない）、各子の `offsetHeight` の最大値を `.rt__detail` の `min-height` に与える（`offsetHeight` は親を `height:0;overflow:hidden` でクリップしても各子の自然高を返すので測定は成立）。**同じ隠しスタックから `.rt__card` の最大 `offsetHeight`（`cardMinH`）と `.rt__lead` の最大 `offsetHeight`（`leadMinH`）も測り**、ライブの側カード／本文段落に `min-height` として与える（＝側カードの箱と、本文段落の下に続く箇条書きの開始位置を全タブで固定）。**ライブのタブを順に切り替えて測るのは不可**＝`<Transition mode="out-in">` の DOM 着脱中に `parentNode` 例外が出る／測定用の `visibility:hidden` フラグが残ると全パネルが消える。測定は **mount 時・`resize`・`document.fonts.ready`・`items` 変更（deep watch）** で再実行。隠しスタックの markup は実 `.rt__detail` と**同一に保つ**（乖離すると測定がズレる＝コメントで明記）。上限キャップは置かない（最も高いタブは現状スライドに収まっているので、その高さを全タブに予約しても安全）。全 reveal 消費者（row=p6／timeline=p12／pair=p18）で**タブ間 `detailH` が一定**かつ overflow0 を measure 確認する。
- **詳細パネルの高さは“最も高いタブ”を下限（`min-height`）に持つ**（§上の安定化）。その下限自体はスライド予算の**内側**なので通常はページングしない＝中身がスライド予算を超えた時だけ `flex` が縮んで **パネル内ページング**にフォールバック（`‹ • • ›` のドット＋前後ボタンをバー右に表示）。ページはブロック境界でのみ分割し、見切れさせない。`relayout()` は `min-height` 適用後に呼ぶ。
- ただし**まず詰め込まない**のが原則。固定高に収めるため: ①冗長な要素は削る（例: 本文と重複するチップは置かない）／②**出典リンクはパネル外の下部バーに常設**（ページ送りで隠さない・縦領域も食わない）。結果、現行デッキは全タブ 1 ページに収まりページャは出ない（保険として常駐）。
- **見出し行は全幅・常に1行**（カード併設パネル `.rt__detail--card` は2行グリッド＝1行目に見出しを `grid-column:1/-1` で全幅、2行目に[本文｜サイドカード]）。見出しを本文カラム（狭い 1fr）に閉じ込めると長い見出し（例:「Context Engineering ＝ 何を知っているか」）が**2行に折り返す**＝**見出しは必ず独立した全幅行**にして1行に保つ（右カードは見出しの下＝2行目へ“配置を下げる”＝ユーザー指摘「右パーツを調整して見出しを1行に」）。単ページ時は2行目左セルの本文を**上揃え**（`--single { justify-content:flex-start }`＝先頭行を見出し直下にピン。本文段落 `.rt__lead` は `leadMinH` を予約して、その下の箇条書きの開始位置もタブ間で固定）、カードは2行目右セルで `align-self:start`（**top ピン**）＋`cardMinH` 予約＋中身 `justify-content:flex-start`（中身ハグではなく**最も高いカード分を予約**＝タブ切替で箱の高さも見出しも動かない／余りは箱の下端に空き。`align-self:center` や中身 `justify-content:center` は再センタリングして“移動”するため不可。下端の hollow は no-shift のため許容＝§163）。`head` を viewport/track の外（`.rt__detail` 直下）に出すので、隠し測定スタックの markup も同じ構造に保つ。
- **タブ切替のトランジションは純粋なクロスフェード＝縦 translate を使わない**（`.rt-enter/leave-*` は `opacity` のみ・`transform: translateY()` 禁止）。**高さを予約して枠を固定しても（§163）、`translateY(±6–8px)` の出入りアニメはパネル箱と側カード（吹き出し `.rt__card`）を毎回上下に“動かして見せる”**＝ユーザーには「同じページ内でタイトル/大パーツの配置がズレる」と映る（**何度も指摘された p13 のズレの正体は settle 時の位置差ではなく、トランジションの動きそのものだった**）。原則＝**動き＝知覚上のシフト**（§5 の「同一ページ内で配置を動かさない」は静止位置だけでなく遷移中の動きにも及ぶ）。さらに `<Transition mode="out-in">` は旧パネルを完全に unmount してから新パネルを mount するため、その隙間で予約 `min-height` を持つ `.rt__detail` が一瞬消え、**枠が潰れる→スライドが再センタリング→タイトルが微動**しうる。予約高さは**決して unmount されない親ラッパー `.rt__stage`**（`min-height: detailMinH`）に**二重に**持たせ、swap 中も枠高を一定に保つ。検証は settle 実測だけで終わらせず、**トランジション中を 30ms 間隔でサンプル**して h1 top・パネル top が**全フレーム一定**であることを measure（settle だけ見ると動きを見逃す＝これが再発の教訓）。
- **高さ予約の測定プローブ（`.rt__measure`）は必ず“レイアウトさせて”測る＝`height:0; overflow:hidden` で潰さない**（何度も再発した p7/p13 タイトル・ズレの、Chromium では出ない最終原因）: 隠し測定スタックを `height:0; overflow:hidden` で畳むと、**Safari/WKWebView（このアプリのキャンバス描画エンジン）は 0 高さのクリップ済みサブツリーのレイアウトを省略しうる**＝`offsetHeight` が 0 で返り、予約する `detailMinH`/`leadMinH`/`cardMinH` が潰れる → 均一化の“床”が消え、**タブ間で中身高が異なるスライドでタイトル・大パーツが動く**（例 p13＝③ Harness は箇条書きが1つ多く 26px 高い／p7＝③ ツール利用はリード・カードが WKWebView では2行に折り返しうる）。Playwright(Chromium) では均一に測れて Δ=0 に見えるため**「Chromium で安定＝WKWebView で安定」ではない**（教訓＝実機キャンバスは WKWebView。engine 非依存にレイアウトを保証するか、少なくとも WKWebView 前提で推論する）。是正＝プローブは **`position:absolute`（フロー外＝実高さでもパネルを押し下げない）＋ `visibility:hidden` ＋ `clip-path: inset(0 0 100% 0)`**（塗り面積0）で隠し、**畳まずに実レイアウトさせて `offsetHeight` を読む**。**⚠️ 測るのは必ず“非スケールの `offsetHeight`”**＝`getBoundingClientRect().height` は Slidev の `transform: scale()` の**スケール後 px** を返すのに CSS `min-height` は**非スケールのレイアウト px**＝混ぜると予約が水増しされる（実測：148→366px に膨張しタイトルが跳ねた）。**⚠️ パネル最高値は“内側の予約（lead/card）を適用した後”に測る**＝自然高さを先に測ると（予約で伸びた実描画より低いので）床が実高より下に落ち均一化に失敗する＝順序は「lead/card を予約 → `nextTick` → detail の最高値を測る」（プローブ側の lead/card にも同じ `minHeight` を bind）。検証は §190 の 30ms サンプルに加え、**予約値（minH）＝実パネル高（非スケール `offsetHeight`）**の一致、かつ**中身高がタブ間で違うスライド（p13）で h1 top が全タブ一定**を measure。
- **サイドカード本文／ノートの改行は文末（。）境界で `<br>`**（§4 のカード改行規律をリビールにも適用）。狭いカード幅で文が2行に折り返す時、放置すると文の途中（例:「…これが第一の／大きな変化。」）で割れる＝**句点（。）の位置に `<br>`** を置き「1文＝1行」に揃える（例:「…も悪化。<br>“良い文脈づくり”が…」「…へ。<br>これが第一の…」「…必須。<br>ただし…」）。各 `<br>` 区間がカード幅に収まることを `Range` 行分割で確認（はみ出すと二次折返し）。単文が長くて1行に収まらない場合のみ自然折返しに任せる（その最終行が短くなるのは可）。**2行に割れる時は均衡させ孤立片を作らない**＝「最適化して／欲しい答えを引き出す」が「…引き／出す」のような孤立尾を作るなら助詞・読点・節境界に `<br>` を置き上下を均す。**小さい補助カード（右下＝大パーツでない側）はサイズ可変**＝無駄な改行が出るなら font/width を詰めて1〜2行に収めてよい（ユーザー指摘「右下の小パーツは適宜サイズ変更し無駄な改行を生むな」）。大きい本文側は据え置く。
- **パネル項目見出しは「大きく・太く・白、重要語だけ着色」**（`RevealItem.head`、`.rt__badge` 1.42rem/800/白）。見出し全体を着色せず、**`<em>` で囲んだ1語だけ**をブランド色にする（`now:true` のタブは purple）。`<span class="rt__en">` は英名・カテゴリの**ミュート従属表示**。番号(①)や年号は**タブ面に任せ、詳細見出しでは省く**。型は「○○ Engineering ＝ <em>意味</em>」（着色＝意味）か「<em>名前</em> 従属語」（着色＝名前）の2系統。

## 6. 配色・アクセントの規律

- **色数を絞る**。ブランドの cyan/purple グラデ以外の色を増やさない。
- 図表は**塗りで強調しない**。強調は線・テキスト・ウェイトで。
- アクセント（グラデ）は見出しキーワード/ラベル/1主役装飾に限定。
- 絵文字を見出しに使う場合は**多色化に注意**（フルカラー絵文字は「色数を絞る」と逆行し AI っぽさ・カジュアル感が出る）。使うなら役割と数を決めて統一する。
- **まとめ／要点スライドは絵文字の“装飾”を避ける**＝箇条書きの各行頭に 🤖🧩🛡️ のような絵文字を散らしても密度は出ず、むしろ「薄い」印象になる。要点は**概念ラベル（mono）＋一文の実体**で構成し、絵文字は**視覚的弁別が要る所だけ**に限定する（落とし穴グリッド `.icell`・行見出し `.row__ico` など）。実例: p25「今日の要点」は行頭絵文字を外し、MCP／Orchestration／Loop の**3本柱**＝`.pillar`（mono ラベル＋一文、主役 Loop だけ purple アクセント）に置換した。
- **アクセントの多色化は“ブランド地続きのランプ”か“意味を持つ役割色”に限る**（§5 のアイコンチップ）＝「色数を絞る」と矛盾させない範囲で**項目別に着色してよい**。許すのは (a) **cyan→indigo→sky→purple の寒色ランプ**（すべてブランドの cyan↔purple の間＝地続き。型一覧など中立な集合の弁別に）と、(b) **役割色**＝amber=注意／rose=危険（落とし穴）・green=良好・purple=現在/行動・cyan=入力/単一・indigo=知識/記憶。**ランダムな虹色は不可**＝色は必ず**順序か意味**を運ぶ（例: 落とし穴 `.igrid` は左 amber／右 rose の暖色＝“ここは危険”の合図、型一覧 `.ptn-grid` は寒色ランプで4型を弁別、Loop は purple＝現在）。これで「殺風景」を解きつつ AI っぽい無秩序な多色化を避ける。
- **暖色アクセントは「殺風景」を解く“意味色”として少量入れる（amber=いま/主役・rose=注意）**（ユーザー指摘「青紫ベースは良いが、重要な箇所やアクセントに赤黄を入れ殺風景にしないで」）＝ブランドの cyan/purple は不変の地のまま、**amber `#fbbf24` ＝「いま／現在／主役」**（このデッキの背骨＝"Loop がいまの主役"という時間軸の主張を運ぶ意味色）と **rose `#fb7185` ＝注意/危険**（落とし穴・短所）を差す。**装飾でなく意味を運ぶ**ので §6「色は順序か意味」に合致し、青紫の地を壊さない。適用は "now/主役" マーカー（`.rt__now` の「いま」ピル・`.now-label`・要点の「← いま」・締めの「いま」/「主役」）と caution（`.igrid` 落とし穴・`.con` 短所・重要語ハイライト `.mk`）に**限定**＝**分量は控えめ（≤~10%・1スライド1語程度）**、撒くと地が壊れる。amber の器: `.hot`（amber-ink 太字）／`.mk`（amber 蛍光アンダーウォッシュ＋白文字・重要語1つ）／`.nowpill`（amber の「いま」ピル）／`.ac-am`・`.ac-ro`（アイコンチップの `--c`）。**⚠️ `.rt__now` の amber ピルは選択タブでも dark-text を保つ**＝`.rt__tab.is-sel .rt__k` が `background-clip:text`（transparent fill）になるので、ピルは `color:#1a1205; -webkit-text-fill-color:#1a1205` で明示的に opt-out（さもないと amber ピルの文字が透明化して消える）。**この暖色を“ブランド違反”として cyan/purple に戻さないこと**（§8.1・剥がすと殺風景へ逆戻り）。
- **同格（並列）のカード／行は「枠の強さ」を揃える＝薄枠と色枠を混ぜない**（ユーザー指摘「p2 のように、枠線が薄いものと強調されているものが混在するのはおかしい」「p5 でなぜ無駄に Remote をハイライトしているのか」）。並列に置いた `.card`／`.row`／`.pillar`／`.flow-step` 等**あらゆる並列パネル**の集合（免責の2枚・fetch/pull/push の3枚・behind/ahead/diverged の3枚・「次の一歩」の2枚・recap「7つの核」の6行・p5「4つのエリア」や walkthrough「4ステップ」のフロー図 等＝**対等な意味のセット**）で、1枚だけ `.accent`／`.warn`／`.b` の**色枠**にし残りを**素の薄枠**（`--line`＝`rgba(255,255,255,.08)`）にすると、「片方だけ強調」に見えて視覚的に不均衡＝どれが主役か誤読させ、素人っぽさが出る。規律：**同一グループの並列要素は枠ウェイトを統一する**（全て色枠 or 全て素）。**意味の差は“枠の有無”ではなく色相で表す**＝全て色枠に揃えたうえで 1枚だけ別色相（brand-b／warn）にして「注意／主役」を示す（例: fetch/pull/push は3枚とも色枠にし push だけ `accent-b`、免責2枚は注意=`accent-b`＋行動=`accent`）。逆に**主役が居ない対等な N 状態**は**全て同じ色相・同じ強さ**（例: behind/ahead/diverged は3枚とも `card accent`）。素の薄枠で揃えるのも可＝アクセントの経済を守りたいフラットな一覧はむしろ全部素にする（例: recap「7つの核」は 1行だけ赤枠だった `.pillar.b` を外し全 6 行を素の薄枠に統一）。**フロー／ステップのチップ図（`.flow-step`）は“静かな地図”＝全ステップを素で揃える**＝パイプラインの終端（p5 の Remote・walkthrough の「④ 反映」）を「主役」扱いして 1枚だけ色枠にしない（4エリア/4ステップは対等な通過点であって climax ではない）。**1枚だけ強調してよいのは “意味のある非対称” に限る**＝before/after の進歩側（§11 の「意味ある色分けセット」＝ 2枚とも色枠で誤解=warn⇄実は=good のように**色相で対比**する形）や、デッキの主張を担う**明示的な唯一の主役**（§6 の要点 3本柱で Loop だけ purple）。**移植時の頻出崩れ**＝ソース（例: AIAgent 免責は両行とも `accent`）から素の `.row`／`.card` に書き換わり薄枠が紛れ込む／流用 CSS の `.pillar.b`・`.flow-step.accent` を1つだけ残す。**⚠️ 監査は `.card`/`.row`/`.pillar` だけでなくデッキ内の“全パネル基底クラス”を洗う**（`.flow-step` の見落とし＝p5 の Remote が生き残った実例）。検証：各並列グループで全要素の**上枠 color のアルファ（＝強さ）が揃うか実測**（薄枠 `.08` と色枠 `.45+` の混在を検出）＋**実寸スクショで「1枚だけ浮く」枠が無いか目視**。
- **新しく足すコンポーネントも“既定のアクセント階層の中だけ”で作る＝便利でも新色相（青など）を持ち込まない**（この回の自己修正：操作スライドの「こうなる」ラベルを一旦 sky-blue `#38bdf8` で作ったが、このデッキの地は amber→orange＝青は地続きでない新色相だった）。役割で色を分けたいときも使ってよいのは**base（amber `#f9a13a`→orange `#f0502f`）＋ gold `#fde047`（重要/結論）＋ rose `#fb7185`（注意/危険）**の範囲＝例「場面／結果」は amber と gold で弁別する（青を足さない）。新ラベルを作る前に「この意味は既存階層のどれか？」を自問する（§6「色は順序か意味／色数を絞る」）。**⚠️ インライン SVG の `<text>` はスライドの巨大な既定 font-size を継承して肥大・見切れる**＝プレゼンテーション属性 `font-size="14"` は継承 CSS に負けるので `.cover__art text { font-size:13px }` のように**CSS で明示**して抑える（表紙グラフのラベルが巨大化した実例）。検証：新コンポーネントの色が base/gold/rose 以外を使っていないか＋SVG ラベルが意図サイズかを実寸で確認。

### 6.1 モーション（動きの規律）

- **モーションは“既に見えている既定”を飾るだけ**＝可視性をアニメに依存させない。`prefers-reduced-motion: reduce`・印刷・ヘッドレス描画では入場アニメが**発火しない**ので、表示を class トリガの遷移に gate すると**空白のまま出る**。CSS `@keyframes` は `@media (prefers-reduced-motion: no-preference)` の**中**に置き、**既定（reduce 側）は完成状態がそのまま見える**設計にする。検証は `element.getAnimations()` で「no-preference 側は入場アニメが付く／reduce 側は空配列」を実測（boot 前の opacity をタイミングでサンプルすると null を返す＝タイミング依存の判定はしない）。
- **入場モーションは“息をする”スライドだけに限定**＝章扉（`.section__eyebrow/h1/.section__lead/.section__chno`）・表紙（`.cover__*`）・締め（`.closing-in > *` を stagger）。中身の詰まった本文スライドは既存のクリック式リビール（RevealTabs）が動きを担うので入場アニメを足さない。**全スライド一律の同じ入場**は「uniform reflex」＝AI っぽさの tell（§8）＝各リビールは中身に合った動きにする。
- **素材は transform＋opacity のみ・ease-out**（`cubic-bezier(.22,1,.36,1)` 等の指数カーブ／bounce・elastic 禁止）。レイアウトプロパティ（top/height/width）はアニメしない。リストは各要素に僅かな delay の stagger まで（それが「意図した動き」）。
- 既存の `nestpulse`（入れ子図リップル）・進化レール矢印スイープと同じく、**新しい動きにも必ず reduced-motion 代替（静止 or crossfade）を書く**（§5 の各 motion と同一ガード）。

## 7. 見出しは「結論型」（結論ファースト / PREP）

- 「何の話か（説明型）」ではなく「**結論・主張（結論型）**」を見出しにする。
- 例: ×「フレームワークについて」 → ○「むかしは“指示”だけ。いま大事なのは“ループ”」
- 章タイトルは短く象徴的に、本文リードで補足。
- 構成も**結論ファースト（PREP法: 結論→理由→具体例→結論）**で並べると、聞き手が迷わない。

### 7.1 構成・章立てと概念の“層”（どこに置くか）

- **トピックは“層”で配置する**。同じキーワードを含むからと安易に近接させない。
  - 例: 「ツール（＝能力そのもの）」は概念の導入章へ。「MCP（＝ツールをつなぐ“規格”・エコシステム）」はインフラ／標準寄りなので、概念導入ではなく**実践・フレームワーク章**に置くと流れが通る。
- **1つのトピックを複数章に分散させない**。関連スライド（例: MCP の“とは”と“エコシステム”）は**隣接**させ、導入 → 応用の順に並べる。
- **同じ内容を複数スライドで重複させない**。詳細比較（表・対話パネル）と「ざっくり選び方」が同じ顔ぶれを繰り返すなら、**どちらか一方に集約**。とくに初心者向けでは “消化しやすい方” を残す（例: 5フレームワークの比較表より、用途別の短い選び方）。「畳んで重複を減らす」（§5）と同じ目的を、対象読者に合わせて判断する。
- **層の違う概念は1行で対比を明記**し、取り違えを防ぐ（重要語1つだけ着色 = §3）。
  - 例: 「ツール＝“能力”そのもの（Function Calling）／MCP＝それらを“つなぐ規格”」。対比は `.note`（フル枠線・淡いティント・サイドストライプ禁止）で1行に収める。
- **対になる操作・概念は、片方だけ出して“違い”を読者の宿題にしない**（ユーザー指摘「fetch と clone の違いの説明がない」）＝混同されやすい兄弟（`clone`↔`fetch`、`fetch`↔`pull`、`merge`↔`rebase` 等）は、一方を教えるスライドで**もう一方との差を必ず1行で明示**する。実例: 「fetch/pull/push」スライドの冒頭に「初回だけ `git clone` ＝ 丸ごと複製（履歴も作業ファイルも `origin` 設定も）／2 回目以降の取得が `fetch`」を置き、**“初回まるごと”対“以降は取得だけ・手元は変えない”** の軸で対比した。新語を出すとき「その語と最も混同される既出語は何か？」を自問し、差分が本文に無ければ**1行の対比（§7.1 の `.note` か lead）を足してから**先へ進む。
- **概念どうしの“関係性”は、関係する概念を最初に説明する場所で言及する**（後出しの独立バナーで急に差分を出さない）。例: 「ツール」を説明するパネル**内**で「ツールが“能力”そのものなら、枠組みを越えて**つなぐ“規格”**が MCP（→ 次ページ）」と橋渡しし、深掘りは次スライドへ送る＝読者は**関係を先に**掴んでから詳細に入れる。両パネルの**後**に独立 `.note` で “ツール vs MCP” を差し込むと「急に出てきた」と読まれる（関係の言及は導入パネルに1行同居 → 次スライドで深掘り、の順が自然）。
- **まだ教えていない用語を“価値判断つき”で先出ししない（怖い/簡単/黒魔術 等の断定は定義の後）**（ユーザー指摘「p6 で rebase を“黒魔術／怖くない”と書くが、初心者はそもそも rebase が何か知らない」）。導入前の章で未定義語に「危険そう／怖くない」等のラベルを貼ると、読者は**知らない語に感想だけ**を渡され迷子になる（“誤解→実は”の対比も、対象を知らなければ刺さらない）。是正：その位置では**全員が既に持つ一般的な不安**へ置き換える（例: rebase→「一度の操作で 壊してしまい そう」／「仕組みが分かれば もう怖くない」＝語を出さず“こわくない”の主題だけ残す）＝固有語の価値判断は**その語を図と本文で定義するスライド（rebase は専用回）まで遅延**する。原則：あるスライドで語を使うなら「この語はここまでに定義済みか？」を自問し、未定義なら**語を消すか一言の定義を同居**させる（§245 の兄弟対比と同根＝読者の前提を先回りしない）。
- **タイトルは本文の“主役語”を含める／口語の指示語（「〜のやつ」）を避ける／コマンド表記を統一する**（ユーザー指摘「p12 は clone の話なのにタイトルに clone が無い・`git clone` と `fetch` で git の有無が揺れる」「p13 の『…のやつ』では分からない」）。(a) **タイトル＝本文が最初に立てる主役**＝本文冒頭が `git clone` を主役にするならタイトルも `clone / fetch / pull / push` と名指す（見出しと中身の約束を一致＝§247）。(b) **口語の指示語を残さない**＝「`Your branch is behind …` のやつ」のような投げやりな deictic は未完成に見える→「`git status` でよく見る、あの状態です」と**対象を名指す**。(c) **同一文/デッキ内でコマンド表記を統一**＝`git clone` と裸の `fetch` を並べない＝プロース中のコマンド参照は**常に `git xxx` 接頭辞**に揃える（`fetch`→`git fetch`）。検証：`grep` で裸のコマンド語（`git ` を伴わない `fetch`/`pull`/`push`）が本文に残っていないか＋タイトルに本文主役語が含まれるかを確認。
- **スライドを動かしたら章タイトル/リードも追従しているか確認**。章の内訳が変わったのにタイトルが古いと、約束と中身がズレる。
- **章が痩せたら独立章を畳む**。スライドを削って1章の中身が薄く（目安: 章扉＋1〜2枚）なったら、その章扉だけを残さず**“層”の近い隣接章へ統合**する（全章に章扉がある中で1章だけ薄い扉は不整合）。例: 「フレームワーク landscape」章を解体し、MCP・選び方を「作るときの勘所」へ同居。**統合時は一括で繰り上げ／更新**＝ `section__chno[data-number]`・目次の項目番号と「全N章」表記・統合先章扉の文脈ラベル／リード／routeまで揃える（残骸の章番号や古い参照を残さない）。**畳んだ章だけが導入していた“用語”も前提にしない**＝章を消すと、その語が説明なしの前提として残る。残すなら一言の定義を添えるか、既出の語に置換する。
- **要点（RECAP）スライドは「いまの主流」を具体で締める**＝定義の一般論（例:「エージェント＝LLM＋ツール＋ループ」）の再掲だけでは弱く、ユーザーに「薄い」と映る。デッキで実際に教えた“現在の中心”＝**道具の標準接続（MCP）／複数の編成（オーケストレーション）／反復の制御（Loop Engineering＝終了条件・ログ・ガードレール・評価）**を3本柱として明示し、「むかしは“指示”ひとつを工夫すればよかった→いまは“仕組み”全体を設計する時代（その最前線が Loop）」の物語に着地させる（p25／クロージング）。各柱は**1行で実体を言い切り**、主役（Loop）だけアクセント＋「← いま」。`Prompt→Loop` の帯は要点側に“進化の背骨”として残す。**表紙（cover）には道筋・結論を前出ししない**＝表紙は問い（タイトル）だけを立て、`Prompt→Context→Harness→Loop` の帯や「いまは “Loop” が主役」の断定は**載せない**（ネタバレ回避＝本編の全体地図 p11–12 と要点 p25 で初めて明かす）。表紙は SERIES／タイトル／サブ／著者に絞り、フッタは下端固定（`.cover__foot { margin-top:auto }`）。帯の重複を“表紙にも要点にも”ではなく**本編・要点のみ**に寄せる。
- **デッキ最大の結論は“作り手の内部メカニズム”ではなく、聞き手に残る“普遍のメタ教訓”にする（表題の問いに答える）**（ユーザー指摘「“ループを回し切る”がこのプレゼンの結論でいいのか」）＝結論スライド（要点＋クロージング）の**最大級の文字**が担うのは、**デッキのタイトルが立てた問いへの答え**であるべき。実例: 表題「AIエージェント、いま何が起きている?」に対し、答えは「**作り込む場所が“指示ひとつ”から“仕組み全体”へ移った**」＝聞き手が明日も使えるメタ教訓。特定の手法名スローガン（“ループを回し切る”＝作り手向けの内部作法）を**最大の主張に据えない**＝入門デッキの締めが専門操作の標語だと、非専門の聞き手には刺さらず、デッキ本来の背骨（“作り方が外側へ進化：Prompt→Context→Harness→Loop”）も埋もれる。**強調の階層＝メタ教訓＞メカニズム**：メカニズム（Loop）は消さず「その**最前線**が Loop」と**従属の1行**（`.tk muted` 等）に落として“現在の焦点”として残す（主役の bigstate ＋従属の補足、の二段構え）。**要点スライドとクロージングの最大文言は同一の結論で揃える**（デッキ内で結論がブレない＝片方だけ旧スローガンが残る状態を作らない）。原則：最大級の文字を置く前に「これはタイトルの問いへの答えか？ 聞き手に残る普遍か、作り手の作法か？」を自問する。
- **シリーズ共通の「免責（Disclaimer）」を表紙直後（開幕2枚目）に固定で1枚置く**＝趣旨は2点。①**誤りがある可能性**（入門向けにかみ砕くため正確さより分かりやすさを優先・AI 分野は進化が速く内容はある時点のスナップショット）②**最後は一次情報で確認**（重要な判断の前に公式ドキュメント・論文などの出典へ／各スライドの出典＋巻末 References へ誘導）。締めは「このシリーズは“迷子にならない地図”、正確な現在地は一次情報で」。**タイトルは「※ 免責事項 Disclaimer」のように注釈記号 ※ を冠して“注釈”感を出す**（`.kome`＝muted・小さめ・わずかに上付きの参照マーク。“ご利用にあたって”のような婉曲表現より「免責事項」と直截に。和文＝白／英語＝`.grad` の二言語見出しは踏襲）。体裁はゴールスライドと同じ `.stack`＋`.row`×2＋`.tk concl` で**開幕2枚（免責→ゴール）の視覚リズムを揃える**。**行見出しは短い“ラベル”**（例:「誤りがある可能性」「一次情報で確認を」＝狭い `.row__head` 列に1行で収まる長さ。文章をタイトルに詰めると語中で割れる＝NG）、本文は §117 どおり各 li を1行に収める。出典免除のメタ系に含める。
- **時事性のあるデッキは“基準日（as of）”を明示する**（内容は必ず古くなる＝読者が鮮度を自分で判断できるように）＝表紙の mono タグに `AS OF YYYY.MM` を添え（`.cover__num` を流用＝新 CSS 不要）、免責の本文は「内容は **YYYY年M月時点** の**スナップショット**」と**具体的な年月**で書く（「ある時点の」等のぼかしにしない）。置くのは**1デッキ1〜2箇所（表紙＋免責）**に留め、各スライドには撒かない（撒くと足場化＝§8）。§219 の抽象な「スナップショット」注意書きと、表紙・免責の**具体年月**を一致させる。
- **「次の一歩」には“追うべきデータソース”を具体名で置く**＝「公式ドキュメントを一次情報に」等の一般論ではキャッチアップの手がかりにならない。**継続的に追う一次ソースを実名＋ドメイン**で挙げる（例: Anthropic Engineering／OpenAI の技術記事・**MCP 公式** modelcontextprotocol.io・**arXiv**＋**Lil'Log**）。各行は「一次情報／プロトコル／研究→要点」の枠で**役割（なぜ見るか）＋実名リンク**を1行に収め、リンクは `a.doc`（ブランド cyan＋下線・`target=_blank`）で巻末 References と URL を整合。
- **新トピック（とくに“旬”の固有手法）は、前スライドの締めで“予告”してから出す**（ユーザー指摘「p18 Rubberduck が突如挿入され、全体の流れからのストーリーが不明」）＝章内で唐突に固有名（Rubberduck 等）が出ると「急に差し込まれた」と読まれる。前スライドの `.tk concl` の**下**に `.nextcue`（前振りフック＝**中央フェードのソリッド・ヘアライン区切り**＋muted・concl より小さい**従属**行）を置き、**次スライドの“主題”を名指しで予告**する（実例: /17 ReAct の締めの下に「ただし同じ頭ひとつでは"自己採点"が甘い。次は**別のAIに"壁打ち"**させて輪の質を上げる」＝/18「Rubberduck：別のAIに壁打ちさせる」の主題を先に植える）。次スライド側でも導入 `.note` で「いまの弱点＝…」と受けて**伏線を回収**する（予告と回収の対で“計画された展開”にする）。`.nextcue` は concl と見分くよう muted＋区切りヘアラインで従属させ、予告語は amber `.go`（next も "now/主役" の温度＝§6）。**⚠️ 区切り線は「点線」を使わない＝図（ダイアグラム）内のフロー線と視覚言語を分ける**（ユーザー指摘「p17 で図内の"再試行"の点線と、"ただし、同じ頭…"の上の区切り点線の区別がつきづらい」）＝**原則: 点線＝図の中のフロー／ループバック／条件分岐に予約**（矢印・ラベルを伴う＝AgentLoop の紫破線「再試行」）**し、構造的なセクション区切りは別言語にする**（中央フェードのソリッド・ヘアライン＝`::before` で `linear-gradient(90deg, transparent, brand-a 38%, transparent)`・幅 `min(48%,21em)`・矢印なし・短め中央）。**同一スライドに点線が2本並ぶと観客は両方を「図の一部（コネクタ）」と誤読する**（hover できない静止状態で判断＝§150 のアフォーダンス原則と同根）。区切り＝ソリッド・フェード／フロー＝ダッシュ＋矢印＋ラベル、と**線種・端形・ラベルの有無で役割を分離**する。**乱発しない**＝毎スライドに付けると eyebrow/番号足場と同じ「定型足場」化する＝**物語のギャップが実際にある継ぎ目にだけ1つ**置く（§8「順序が情報を持つ時だけ」に準ずる）。この“予告→回収”は改行 QA の対象容器にも含める（`.nextcue` は `<div>`＝§4 auto-phrase 個別付与）。
- **“トレンド”も載せる。ただし通称が一次情報で裏取りできなければ、正確な意味に置き換えてから本物の手法へ橋渡しする**（“情報弱者にならない”＝基礎だけでなく旬の手法もキャッチアップさせたい）。コミュニティで通る通称（例: ユーザー要望の「Rubberduck＝複数モデルを組み合わせる手法」）が**一次情報で確認できない／意味がズレている**ことがある。そのときユーザーの言い分を“事実”として載せない＝**(1) その語の正確な意味を確認**（Rubberduck＝AI に説明して気づく“ラバーダック・デバッグ”＝壁打ちの比喩。「複数モデル結合」の定義は出典なし）、**(2) 認知される通称はタイトルに残す**（探しやすさ・要望尊重）、**(3) 中身は一次情報のある実在手法に橋渡し**（複数モデル系なら LLM-as-a-Judge〈arXiv:2306.05685〉／Architect-Editor〈aider.chat〉／Mixture-of-Agents〈arXiv:2406.04692〉等を各カードに出典付きで）。**捏造した・名前と食い違う出典を貼った“named technique”として出さない**。前提が怪しい用語は**まず research で一次情報を当たり、決め手がなければユーザーに枠組みを確認**してから作る（本件は確認のうえ「名前は残す・意味は正確に・実手法へ橋渡し」を選択）。

## 8. 避けること（AI っぽさ・スロップ回避）

- 文字だらけのスライド / 読まないと理解できない構成
- 抽象的すぎる見出し / 汎用ビジネス素材感
- **意味のない装飾**: 全面グラデ、過剰な glow/影、全要素が同じ角丸、タイトル下のアクセント下線
- 役割ごとの文字サイズ・太さが**スライドごとにばらつく**こと
- AI が罫線・区切り線・図版を**新規に量産**すること（既存のトーンを崩す）
- 低コントラストの muted 文字 / プレースホルダの残り / 見切れ・重なり・折返し崩れ
- 並列・同種の要素のうち**1つだけを塗り/グロー/サイズ/字体で過剰強調**すること（意図した主役以外が目立つ）。とくに同心（入れ子）図は内側ほど強調になりがちで、**物語の主役（多くは外側＝到達点）と矛盾**しないか確認する。強調は「意図した1要素を1手段（着色など）で」が原則。
- **原稿との対応を一文で説明できない図を「meaningful visual」と呼ばない**。S字線・波形・単独アイコン・`?`や一文字略号のように、凡例なしでは「何と何を、どの順で、なぜ結ぶか」が読めない図は装飾であり不合格。比較なら両側の実項目と比較軸、工程なら実項目を置いたnodeと方向、分岐なら選択肢を日本語で明示する。線は実nodeを接続し、矢印・段階名・到達点のいずれかを運ぶ時だけ残す。**本文を図の雰囲気へ要約せず、図そのものを原稿の読み順にする**。canonical 16:9と変換後を同じcontact sheetで照合し、各図形がどの原稿項目を表すか説明できなければ修正する（教訓＝Reader pilotでcoverage数は満たしたが、抽象線画と浅い連結文では「元スライドの内容が消えた」と読まれた）。
- **語句のcoverageと、意味関係のcoverageを混同しない**。atomic itemが100% DOMに存在しても、比較を一本道へ、入れ子を時系列へ、複数の一致証言を別ページへ、全体で読む工程表を途中分断へ変えると原稿の主張は失われる。portrait変換では、比較＝両側＋共通結論、入れ子＝包含関係、引用群＝同時に見える共通主張、工程表＝全工程を見渡せるoverviewというcanonicalの**関係構造**を同じviewportに残す。同じrail・丸番号・下端takeawayを意味の異なるsourceへ反復しない。presenter notesのうち、本文の読み方・例・因果を成立させる説明はvisible editorial copyであり、補足dialogへ退避してはならない（dialog-onlyは出典・追加情報）。contact sheetの縮小像だけで承認せず、390×844実寸で「図を隠しても説明が通るか／図が文章より一段早く関係を伝えるか」をsourceごとに確認する。
- **縦型化そのものを完成条件にしない**。完成済み16:9の比較軸・入れ子・引用群・工程表をportraitへ再編集するほど意味関係や視覚品質が落ちる場合、変換を続けるのは不合格＝**canonicalをそのまま保ち、閲覧操作をスマホへ適応する**。実テキスト／SVG／RevealTabs／Citeを保持した980×552 HTMLを全体表示し、ピンチ・ダブルタップ・明示ボタンで拡大、拡大中はパン、横向きは表示領域を拡張する。全画面画像へ置換して検索・選択・操作を失わない。全体表示の上下に巨大な黒帯を残すPDF viewer模倣も避け、16:9面は内容高にhugさせ、拡大時だけパン領域を全高へ展開する（教訓＝portrait Readerを反復改善してもcanonicalより悪化し、ユーザーが横型保持を選択）。
- **装飾（マスコット・バッジ等）が、反復カードの“構造行”を動かして整列を崩す**こと（ユーザー指摘「p16 の X いいねアイコンが上下パネルでずれている」）＝同じカード部品を縦積み/反復するとき、片方だけに装飾を足すとその装飾ぶんの右インセットがカード全体に乗り、**フッタのアクション列・ヘッダのロゴ（＝メタ/chrome 行）が隣のカードと横位置でずれる**。観客は反復要素の「同じ位置にあるはずの目印」が揃わないと即座に気づく。**原則: 装飾は“ガター（余白）を占有するだけ”に留め、本文（可変長テキスト）と chrome 行（ロゴ・アイコン列）は装飾の有無に関わらず同じ基準線で揃える**。実装は「装飾カードは箱ごと `--xp-gutter` ぶん右インセット、装飾なしカードは**chrome 行だけ**同じインセット・本文は全幅のまま（再ラップさせない＝高さも変わらない）」。この“片側だけ inset”は**装飾カードを含むフィード内に限定**して掛ける（`.xfeed:has(.x--deco) .x:not(.--deco) …`＝`:has()` で限定）＝単独で再利用した装飾なしカードに**幽霊ガター**を出さない（部品の可搬性を壊さない）。ガター定数はカード側の CSS 変数に一元化し、フィード側の整列規則はその変数を継承して使う（値の二重管理を避ける）。
- ＞ 章番号（01/02…）やセクション上の eyebrow を「全スライドの定型足場」にすること。
   本デッキは既存ブランドとして採用済みだが、**新規に作るなら“意味のある所だけ”**に絞るのが原則。

## 9. ゼロから作るための指示テンプレート（コピペ用）

AI（とくに Claude）に渡すときの**4つのコツ**（kappa 記事）:
1. **役割を与える** — 「あなたは外資系コンサルのシニアコンサルタント兼プレゼンデザイン専門家です」。
2. **4大原則を明示** — 近接・整列・コントラスト・反復を指示に入れ、「どこを強調するか」を書かせる。
3. **XML タグで構造化** — `<role>` `<instructions>` `<rules>` `<context>` で指示と素材を分ける（Claude に特に有効）。
4. **肯定形で書く** — 「専門用語を使わないで」ではなく「中学生でも分かる言葉で説明して」。

### 9.1 構造化テンプレート（Claude 向け・推奨）

```text
<role>
あなたは外資系コンサルのシニアコンサルタント兼プレゼンデザインの専門家です。
ダーク×cyan/purple ブランドの Slidev デッキを、規律あるデザインで作成・改善します。
</role>

<instructions>
以下の <context> を基に、スライドデッキ（全【N】枚）の構成案とスライドを作成してください。
出力は slides.md（Markdown＋最小限のHTML）と style.css で行い、<rules> を厳密に守ってください。
<brand> は変更せず、規律だけを底上げします（作り直しではなく規律の上書き）。
</instructions>

<rules>
1. 結論ファースト（PREP法）。1スライド＝1メッセージを徹底し、情報量を絞る。
2. デザインの4大原則（近接・整列・コントラスト・反復）を守り、各スライドで強調箇所（太字/グラデ/サイズ）を明記する。
3. 役割ごとに文字サイズ・太さを全スライドで固定（タイトル≒本文の2倍、本文は最低1rem）。
4. 本文・箇条書きは左揃え、中央寄せは表紙/章見出しのみ。グラデ文字は見出しキーワード/ラベルだけ。
5. 全スライドに視覚要素（図/カード/チップ/タイムライン/表/アイコン）を1つ以上。レイアウトは使い回さず変化させる。
6. 専門用語は避け、中学生でも直感的に分かる平易な言葉で書く（肯定形で指示）。
7. 余白は外周3rem、間隔は0.5/1/1.5remの3段。日本語の折返しは em で測り、word-break:auto-phrase を使う。
</rules>

<brand>
・背景: ダーク #07080d→#0b0d15 ＋四隅に cyan #22d3ee / purple #a855f7 の radial、微細グリッド。
・アクセント: cyan→purple グラデ。フォント: 見出し・本文 Noto Sans JP / ラベル・数値・コード JetBrains Mono。
・配色・ロゴ・トーンは固定（変更禁止）。
</brand>

<context>
・テーマ:【例: AIエージェント入門】
・対象読者:【例: 初心者】
・ゴール（最後に取ってほしい行動）:【…】
・含めたい情報:【メモ/箇条書きで素材を投げる】
・章立ての軸:【例:「昔はこうだった→いまはこう」という変遷】
</context>

<output>
最後に各スライドを PNG/PDF 化して目視QA（重なり/見切れ/余白不足<48px/カラム不揃い/低コントラスト/折返し崩れ/プレースホルダ残り）。問題→修正→再確認を最低1周。
</output>
```

## 10. 実装メモ（このリポジトリ固有）
